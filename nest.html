Nerd-Nest

1. Introduction to NestJS

  • Framework for building scalable, maintainable Node.js server apps
  • Built with TypeScript, uses modern JavaScript features
  • Inspired by Angular (modular design, decorators, DI)
  • Helps organize code for large applications
  • Suitable for enterprise-level backend projects

  • Built-in architecture: Modules, controllers, providers ready out of the box
  • TypeScript-first: Strong typing and better tooling
  • Dependency Injection: Simplifies managing dependencies
  • Extensible: Can run on Express or Fastify server engines
  • More features: Validation, guards, interceptors, middleware included
  • Better testing support: Modular design aids unit/integration testing

  • Module: Groups related components, e.g., AppModule
  • Controller: Handles HTTP requests and routes
  • Provider (Service): Contains business logic, injected where needed
  • Dependency Injection: Inject dependencies automatically for loose coupling
  • Middleware: Functions before route handling (e.g., logging, auth)
  • Guard: Protects routes (authorization)
  • Pipe: Validates/transforms input data
  • Interceptor: Extends method behavior (logging, caching)

  • Install CLI globally:

    npm install -g @nestjs/cli

  • Create new project:

    nest new project-name

  • Generate components easily:

    nest generate module users
nest generate controller users
nest generate service users

  • CLI auto-creates project with TypeScript, config, and basic structure.

    src/
├── main.ts              # Entry point, bootstraps app
├── app.module.ts        # Root module
├── app.controller.ts    # Root controller (handles routes)
├── app.service.ts       # Root service (business logic)
└── users/               # Example feature folder
    ├── users.module.ts
    ├── users.controller.ts
    └── users.service.ts
  • Organized by features/modules.
  • Each module encapsulates controllers and providers.
  • main.ts starts the app with NestFactory.create(AppModule).

2. Core Concepts

  • Fundamental building blocks of a NestJS app
    • Organize related controllers and providers
    • Every app has at least one root module (AppModule)
    • Decorated with @Module()
    • Can import/export other modules to share providers

      • @Module({
    imports: [],
    controllers: [UsersController],
    providers: [UsersService],
  })
  export class UsersModule {}

  • Handle incoming HTTP requests and return responses
  • Define routes using decorators like @Get(), @Post()
  • Inject services/providers to process logic
  • Decorated with @Controller('route')

    • @Controller('users')
export class UsersController {
  constructor(private readonly usersService: UsersService) {}
                  
  @Get()
  findAll() {
    return this.usersService.findAll();
  }
}

  • Classes that can be injected via Dependency Injection
  • Used to encapsulate business logic, database calls, etc.
  • Decorated with @Injectable()
  • Services are a common type of provider
  • Registered in module providers array

  • A specific kind of provider focused on business logic
    • Handles data processing, interacts with database or APIs
    • Injected into controllers or other providers

    @Injectable()
export class UsersService {
  findAll() {
    return ['user1', 'user2'];
  }
}

  • Design pattern where dependencies are provided rather than created internally
  • Promotes loose coupling and easier testing
  • NestJS injects dependencies automatically when declared in constructor
  • Uses metadata and decorators to resolve dependencies

  • Define providers with custom injection tokens or factory functions
    • Useful for injecting values, classes, or async providers
    • Types: Class provider, value provider, factory provider, existing provider
  • Example (factory provider): 
    {
  provide: 'CONFIG',
  useFactory: () => ({ port: 3000 }),
}

  • Special methods called at specific points of provider lifecycle
  • Common hooks:
    • onModuleInit() — runs after module initialization
    • onApplicationBootstrap() — runs after app bootstrap
    • onModuleDestroy() — runs before module is destroyed
    • onApplicationShutdown() — runs on app shutdown

  • Implement by adding interface and method in provider/service:

    export class UsersService implements OnModuleInit {
  onModuleInit() {
    console.log('Module initialized');
  }
}

3. Routing

  • Routes are defined inside controllers using method decorators.
  • Common decorators: @Get(), @Post(), @Put(), @Delete(), @Patch()
  • Decorate methods with HTTP method and optional path

  • Example:

    @Controller('users')
export class UsersController {
  @Get()
  getAllUsers() {
    return [];
  }
                  
  @Post()
  createUser() {
    return 'User created';
  }
}

  • Route parameters: Dynamic parts of URL (e.g., /users/:id)
  • Access using @Param() decorator
  • Example: 
    @Get(':id')
getUser(@Param('id') id: string) {
  return `User ${id}`;
}
  • Query parameters: Key-value pairs after ? in URL (e.g., /users?role=admin)
  • Access using @Query() decorator
  • Example: 
    @Get()
getUsers(@Query('role') role: string) {
  return `Role: ${role}`;
}

  • Access the HTTP request body (usually JSON) with @Body() decorator
  • Used mostly with POST, PUT, PATCH requests
  • Can use DTOs (Data Transfer Objects) to validate/define request shapes
  • Example: 
    @Post()
createUser(@Body() createUserDto: CreateUserDto) {
  return createUserDto;
}

  • GET: Retrieve data
  • POST: Create new data
  • PUT: Update existing data fully (replace)
  • PATCH: Update existing data partially
  • DELETE: Remove data

  • Nest provides decorators for each method: @Get(), @Post(), @Put(), @Patch(), @Delete()

  • Routes can have dynamic segments with :param syntax
  • Multiple parameters possible (e.g., /users/:userId/posts/:postId)
  • NestJS extracts them with @Param()
  • Useful for resource-specific endpoints
  • Example: 
    @Get(':userId/posts/:postId')
getPost(@Param('userId') userId: string, @Param('postId') postId: string) {
  return `User ${userId}, Post ${postId}`;
}

  • Control access to routes for authorization/authentication
  • Implement CanActivate interface
  • Return true to allow access or false to block
  • Use @UseGuards() decorator to apply guards to controllers or routes
  • Example guard usage: 
    @UseGuards(AuthGuard)
@Get('profile')
getProfile() {
  return 'User Profile';
}

4. Middleware

  • Middleware are functions executed before route handlers. Used for logging, authentication, request modification, etc.
  • Implement the NestMiddleware interface with a use(req, res, next) method.
  • Example: 
    
import { Injectable, NestMiddleware } from '@nestjs/common';
                  
@Injectable()
export class LoggerMiddleware implements NestMiddleware {
  use(req: Request, res: Response, next: Function) {
    console.log(`Request... ${req.method} ${req.url}`);
    next();
  }
}

  • Globally: Applied to all routes using app.use() in main.ts. 
    
app.use(LoggerMiddleware);
  • Per route or module: Use configure() method in a module implementing NestModule. 
    
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes('users'); // applies only to /users routes
  }
}

  • Built-in middleware: Provided by NestJS (e.g., Logger, Helmet, Cors).
  • Custom middleware: User-defined, tailored for specific app needs (e.g., custom logging, auth checks).
  • Built-in middleware are easy to use and configure; custom middleware gives full control.

  • NestJS can use Express or Fastify middleware. Use .apply() method in configure() to integrate.
  • Example integrating helmet middleware: 
    import * as helmet from 'helmet';
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    consumer
      .apply(helmet())
      .forRoutes('*');
  }
}

  • Enables use of popular middleware libraries easily.

5. Exception Handling

  • NestJS provides common HTTP exceptions extending HttpException.
    • NotFoundException (404)
    • BadRequestException (400)
    • UnauthorizedException (401)
    • ForbiddenException (403)

  • Throw these in controllers/services to send proper HTTP error responses.

  • Example: 
    throw new NotFoundException('User not found');

  • Catch and handle exceptions thrown anywhere in the app globally.ß
  • Implement ExceptionFilter interface and use @Catch() decorator.ß
  • Register globally using:ß 
    app.useGlobalFilters(new AllExceptionsFilter());

  • Useful for consistent error formatting and logging.

  • Create filters to handle specific exceptions or customize responses.
  • Can catch multiple or single exception types via @Catch(ExceptionType).
  • Access exception, request, response objects to tailor handling.
  • Example: 
    @Catch(HttpException)
export class HttpExceptionFilter implements ExceptionFilter {
                    catch(exception: HttpException, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const status = exception.getStatus();
                  
    response.status(status).json({
      statusCode: status,
      message: exception.message,
    });
  }
}

  • Use try-catch blocks inside async methods to catch exceptions.
  • Throw exceptions inside catch block for Nest to handle.
  • Example: 
    async findUser(id: string) {
  try {
    const user = await this.userRepository.findById(id);
    if (!user) throw new NotFoundException('User not found');
    return user;
  } catch (error) {
    throw new InternalServerErrorException();
  }
}

6. Pipes

  • Pipes operate on incoming request data before handlers process it.
  • Common built-in pipes:
    • ValidationPipe: validates input data using class-validator decorators.
    • ParseIntPipe: converts string params to integers.
    • ParseBoolPipe, ParseUUIDPipe, etc.: for type parsing.
  • Used via method parameter decorators or globally.
  • Example: 
    @Get(':id')
getUser(@Param('id', ParseIntPipe) id: number) {
  return `User ID: ${id}`;
}

  • Implement PipeTransform interface with transform(value, metadata) method.
  • Can transform, validate, or reject data.
  • Decorate class with @Injectable().
  • Example: 
    @Injectable()
export class MyCustomPipe implements PipeTransform {
  transform(value: any) {
    // custom validation or transformation
    return value;
  }
 }

  • Pipes can modify input data (e.g., convert string to int) before it reaches the route handler.
  • Validate input and throw exceptions if invalid.
  • Can be applied:
    • Per-route or method parameter:

  • @Post()
createUser(@Body(new ValidationPipe()) createUserDto: CreateUserDto) {}
  • Globally using app.useGlobalPipes()
  • Enhances security and data consistency

7. Guards & Authorization

  • Guards control access to routes by implementing the CanActivate interface.
  • canActivate(context: ExecutionContext) method returns true (allow) or false (deny).
  • Apply guards using the @UseGuards() decorator on routes or controllers.
  • Example: 
    @Injectable()
export class AuthGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const request = context.switchToHttp().getRequest();
    return Boolean(request.user); // allow if user exists
  }
}

  • Control access based on user roles or permissions (e.g., admin, user).
  • Roles assigned to users determine their permissions.
  • Use guards to check roles before allowing access.
  • Often combined with decorators like @Roles('admin') for cleaner code.
  • Example guard checking roles: 
    canActivate(context: ExecutionContext): boolean {
  const request = context.switchToHttp().getRequest();
  const user = request.user;
  return user?.roles.includes('admin');
}

  • Authentication: Verifies user identity (e.g., login, JWT verification).
  • Authorization: Determines if authenticated user has permission to access resources (e.g., RBAC).
  • Authentication is usually handled by strategies (Passport.js integration).
  • Authorization is enforced via guards.

  • Guards check if a request is allowed (authorization).
  • Interceptors modify or extend how the request or response is handled (e.g., logging, caching).
  • Can be used together to enforce security and modify behavior.
  • Applied with decorators:
    • @UseGuards(AuthGuard)
@UseInterceptors(LoggingInterceptor)
@Get()
getData() {}

8. Interceptors

  • Intercept incoming requests and outgoing responses for logging purposes.
  • Implement NestInterceptor interface and override intercept() method.
  • Can log method, URL, execution time, etc.
  • Example: 
    export class LoggingInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable {
    console.log('Before...');
    const now = Date.now();
    return next.handle().pipe(
      tap(() => console.log(`After... ${Date.now() - now}ms`)),
    );
  }
}

  • Modify or format the response before sending it to the client.
  • Used to change data structure, remove sensitive fields, or wrap responses.
  • Implement in intercept() and map the result.
  • Example: 
    return next.handle().pipe(map(data => ({ data, status: 'success' })));

  • Catch and transform exceptions or errors thrown in route handlers.
  • Wrap errors into consistent formats or log them.
  • Interceptors can complement exception filters.
  • Example use: transform database errors into HTTP exceptions.

  • Set a timeout limit for request handling.
  • Automatically throw error if handler exceeds allowed time.
  • Example using RxJS timeout operator: 
    return next.handle().pipe(timeout(5000));

  • Cache responses to improve performance.
  • Check cache before handling request, return cached response if available.
  • Save response data to cache after processing.
  • Often combined with Redis or in-memory cache stores.
  • Example: cache interceptor stores frequently requested data.

9. Validation

  • class-validator: library for declarative validation with decorators (e.g., @IsString(), @IsNotEmpty()).
  • class-transformer: transforms plain objects into class instances and supports data transformation.
  • Both integrate seamlessly with NestJS DTOs for validation and transformation.
  • Example: 
    import { IsString, IsInt } from 'class-validator';
                  
export class CreateUserDto {
  @IsString()
  name: string;
                  
  @IsInt()
  age: number;
}

  • Classes defining the shape and validation rules of data sent over the network (request payloads).
  • Used as method parameters decorated with @Body().
  • Improve code clarity, validation, and type safety.
  • Example: 
    @Post()
createUser(@Body() createUserDto: CreateUserDto) {}

  • Always validate incoming data to avoid malformed or malicious input.
  • Use DTOs with decorators for clear, reusable validation rules.
  • Validate both route params and request bodies.
  • Provide meaningful error messages.
  • Keep validation logic separate from business logic.

  • NestJS built-in pipe that automatically validates DTOs on incoming requests.
  • Throws exceptions if validation fails (HTTP 400).
  • Can be applied per-route, per-controller, or globally (app.useGlobalPipes()).
  • Example global usage: 
    app.useGlobalPipes(new ValidationPipe());

10. Configuration Management

  • Official NestJS module to manage app configuration.
  • Loads configuration from .env files or other sources.
  • Provides a ConfigService to access config values anywhere via Dependency Injection.
  • Easy to organize and centralize config management.
  • Setup example: 
    import { ConfigModule } from '@nestjs/config';
                  
@Module({
  imports: [ConfigModule.forRoot()],
})
export class AppModule {}

  • Store sensitive or environment-specific settings outside code (e.g., API keys, DB URLs).
  • .env file contains key-value pairs:
    • 
DATABASE_URL=postgres://user:pass@localhost:5432/dbname
PORT=3000
  • @nestjs/config reads .env automatically with forRoot().
  • Use ConfigService to safely access variables:
    • const port = this.configService.get('PORT');

  • Validate environment variables to avoid misconfiguration at runtime.
  • Use libraries like Joi to define validation schemas.
  • Pass schema to ConfigModule.forRoot() to auto-validate .env contents.

  • Example:

    import * as Joi from 'joi';
                  
ConfigModule.forRoot({
  validationSchema: Joi.object({
    PORT: Joi.number().default(3000),
    DATABASE_URL: Joi.string().required(),
  }),
});

  • Prevents app startup with invalid or missing config values.

11. Database Integration

  • TypeORM: popular ORM for SQL databases (Postgres, MySQL, etc.)
  • Official NestJS package: @nestjs/typeorm
  • Define entities as classes with decorators (@Entity(), @Column())
  • Use TypeOrmModule.forRoot() to configure DB connection
  • Inject repositories/services to perform CRUD operations

  • Example entity:

    import { Entity, PrimaryGeneratedColumn, Column } from 'typeorm';
                  
@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;
                  
  @Column()
  name: string;
}

  • Prisma: modern ORM for type-safe database access, supports SQL DBs
  • Use Prisma Client inside Nest services for queries
  • Schema defined in schema.prisma file
  • Prisma doesn’t have official NestJS module but works well with Dependency Injection
  • Generate Prisma Client via CLI, inject in services
  • Example usage: 
    const user = await this.prisma.user.findMany();

  • @nestjs/mongoose package integrates Mongoose ODM for MongoDB
  • Define schemas using decorators (@Schema(), @Prop())
  • Register models with MongooseModule.forFeature()
  • Use injected models for CRUD

  • Example schema:

    import { Schema, Prop, SchemaFactory } from '@nestjs/mongoose';
                  
@Schema()
export class Cat {
  @Prop()
  name: string;
}
                  
export const CatSchema = SchemaFactory.createForClass(Cat);

  • Abstracts data access logic behind repository classes
  • Used in TypeORM with Repository<T>
  • Helps keep service code clean and focused on business logic
  • Inject repositories via constructor
  • Example: 
    constructor(
  @InjectRepository(User) private userRepository: Repository,
) {}

  • Transactions ensure multiple DB operations succeed or fail together
  • Supported in TypeORM and Prisma
  • In TypeORM, use @Transaction() decorator or queryRunner

  • Example (TypeORM):

    await this.connection.transaction(async manager => {
  await manager.save(user);
  await manager.save(profile);
});

  • Manage schema changes incrementally and safely
  • TypeORM and Prisma provide CLI tools for generating and running migrations
  • Migrations help version control DB schema alongside code
  • Example (TypeORM CLI): 
    typeorm migration:generate -n AddUserTable
typeorm migration:run

12. Authentication

  • Uses JSON Web Tokens (JWT) to securely transmit user identity
  • Server issues JWT after login, client stores and sends it with requests (usually in Authorization header)
  • Stateless: no session stored on server
  • Verify token on each request via guard or middleware
  • Example usage: 
    @UseGuards(JwtAuthGuard)
@Get('profile')
getProfile() {
  return this.userService.getProfile();
}

  • A popular authentication middleware integrated with NestJS
  • Strategies:
    • Local: username/password verification
    • JWT: validate and decode JWT tokens
    • OAuth: third-party providers (Google, Facebook, GitHub)
  • Use @nestjs/passport package to implement strategies
  • Example local strategy: 
    @Injectable()
export class LocalStrategy extends PassportStrategy(Strategy) {
  async validate(username: string, password: string) {
    // validate user credentials
  }
 }

  • Long-lived tokens used to obtain new short-lived JWTs without re-login
  • Stored securely (httpOnly cookies or secure storage)
  • Helps maintain user sessions without compromising security
  • Refresh token endpoint verifies and issues new JWT
  • Important for logout and token invalidation logic

  • Traditional authentication storing session state on server (memory, Redis)
  • Uses cookies to track sessions
  • Less scalable than JWT but easier for some use cases
  • Supported in NestJS via express-session and Passport

  • Example: configure session middleware in main.ts

  • OAuth-based login with third-party providers
  • Use Passport OAuth strategies like passport-google-oauth20
  • Users authenticate via provider; app receives user info and access tokens
  • Enables easy login without password management

  • Example: implement Google strategy and handle OAuth callback routes

13. File Uploads

  • NestJS uses Multer middleware (built-in integration via @nestjs/platform-express) for handling multipart/form-data
  • Use @UseInterceptors(FileInterceptor('fieldname')) to intercept file uploads
  • Access uploaded file with @UploadedFile() decorator
  • Example: 
    @Post('upload')
@UseInterceptors(FileInterceptor('file'))
uploadFile(@UploadedFile() file: Express.Multer.File) {
  console.log(file);
  return { message: 'File uploaded successfully' };
}

  • Save uploaded files to server disk using Multer's disk storage engine
  • Configure destination folder and filename in Multer options
  • Example disk storage config: 
    @UseInterceptors(FileInterceptor('file', {
  storage: diskStorage({
    destination: './uploads',
    filename: (req, file, cb) => {
      const uniqueName = Date.now() + '-' + file.originalname;
      cb(null, uniqueName);
    },
  }),
}))

  • After receiving file, upload it programmatically to cloud storage services
  • Use official SDKs (AWS SDK, Google Cloud Storage client) inside service methods
  • Store URLs or metadata in database for reference
  • Example (AWS S3):

    Receive file in controller, pass buffer or stream to S3 upload method in service, return S3 file URL after successful upload

14. Event-Driven Architecture

  • Built-in event bus for intra-application event handling
  • Uses EventEmitter2 package integrated via @nestjs/event-emitter
  • Emit events and subscribe with listeners
  • Example: 
    // Emit event
this.eventEmitter.emit('user.created', payload);
                  
// Listen for event
@OnEvent('user.created')
handleUserCreatedEvent(payload: any) {
  // handle event
}

  • Used for decoupled, scalable communication between microservices or distributed components
  • NestJS microservice package integrates RabbitMQ, Kafka, NATS
  • Supports publish-subscribe, message routing, durability
  • Use transport layers like ClientsModule.register() with config
  • Example with RabbitMQ: 
    ClientsModule.register([
  { 
    name: 'RABBITMQ_SERVICE', 
    transport: Transport.RMQ, 
    options: { 
      urls: ['amqp://localhost'], 
      queue: 'main_queue' 
    }
  }
]);

  • Real-time bidirectional communication protocol
  • Supported via @WebSocketGateway() decorator
  • Handle events with @SubscribeMessage()
  • Used for chat apps, live notifications, etc.
  • Example: 
    @WebSocketGateway()
export class EventsGateway {
  @SubscribeMessage('message')
  handleMessage(client: any, payload: any) {
    return 'Hello world!';
  }
}

  • Microservices communicate via events/messages instead of direct HTTP calls
  • Use NestJS microservices package with transporters (TCP, Redis, NATS, RabbitMQ)
  • Supports request-response and event-driven patterns
  • Promotes loose coupling and scalability
  • Example: 
    const app = await NestFactory.createMicroservice(AppModule, { transport: Transport.TCP });
await app.listen();

15. REST API Development

  • Manage multiple API versions without breaking existing clients
  • NestJS supports versioning via URL, header, or media type
  • Enable versioning in main app module:
    • app.enableVersioning({
  type: VersioningType.URI, // e.g., /v1/users
});
  • Use @Version('1') decorator on controllers or routes

  • Handle large datasets by returning data in chunks (pagination)
  • Use query parameters for pagination (page, limit) and filtering (status, category)
  • Implement logic in service layer to fetch subset of data
  • Example query params:
    • GET /users?page=2&limit=10&role=admin

  • REST principle where responses include links to related resources or next actions
  • Helps clients navigate API dynamically
  • Example: returning a user with links to edit or delete
    • {
  "id": 1,
  "name": "John",
  "_links": {
    "self": "/users/1",
    "edit": "/users/1/edit",
    "delete": "/users/1"
  }
}

  • NestJS integrates Swagger via @nestjs/swagger
  • Automatically generate interactive API docs
  • Decorate controllers, DTOs, and methods for rich docs
  • Setup example:
    • const config = new DocumentBuilder()
  .setTitle('API')
  .setVersion('1.0')
  .build();
const document = SwaggerModule.createDocument(app, config);
SwaggerModule.setup('api', app, document);

  • Customize API responses by transforming data
  • Remove sensitive fields, add metadata, or format output
  • Use interceptors or DTO transformation
  • Example with interceptor to wrap response:
    • return next.handle().pipe(map(data => ({ status: 'success', data })));

16. GraphQL

  • A query language for APIs and runtime for executing queries
  • NestJS supports GraphQL via @nestjs/graphql package
  • Enables flexible data fetching and avoids over/under-fetching
  • Integrate by importing GraphQLModule in app module
  • Example:
    • GraphQLModule.forRoot({
  autoSchemaFile: true, // code-first schema generation
});

  • Schema-first:
    • Write GraphQL schema (.graphql files) manually
    • Generate resolvers based on schema
    • More control over schema but extra maintenance
  • Code-first:
    • Define schema using TypeScript decorators (@ObjectType(), @Field())
    • Schema is generated automatically from code
    • Faster development with type safety and easier refactoring
  • NestJS supports both, but code-first is more common in Nest

  • Queries: Read operations, fetch data
  • Mutations: Write operations, create/update/delete data
  • Subscriptions: Real-time data updates over WebSockets
  • Define resolvers for each type
  • Example query resolver:
    • @Query(() => [User])
  getUsers() {
    return this.userService.findAll();
  }

  • Batch and cache requests to avoid redundant database calls
  • Useful in GraphQL to solve the “N+1” problem
  • DataLoader groups multiple requests into a single batch request
  • Integrate by creating a DataLoader instance per request and injecting it into resolvers
  • Improves performance and reduces DB load

18. Security

  • Trick users into submitting unauthorized requests
  • Use CSRF tokens to validate legitimate requests
  • NestJS apps with sessions can use csurf middleware for CSRF protection
  • For stateless APIs (JWT), CSRF risk is lower but consider same-site cookies and CORS policies

  • Protect APIs from abuse and DoS attacks by limiting request rate
  • Use NestJS @nestjs/throttler package for easy rate limiting
  • Example setup:
    • ThrottlerModule.forRoot({ ttl: 60, limit: 10 }); // 10 requests per minute
  • Apply globally or per-route with guards

  • Clean incoming data to remove malicious input (e.g., scripts, HTML tags)
  • Prevent XSS and injection attacks
  • Use libraries like class-sanitizer alongside class-validator
  • Sanitize user inputs before processing or saving

  • Use middleware like helmet to set HTTP headers for security (e.g., CSP, XSS protection)
  • Easy integration in NestJS:
    • app.use(helmet());
  • Helps prevent attacks like clickjacking, MIME sniffing, and more

  • Avoid building queries with raw user input strings
  • Use parameterized queries or ORM query builders
  • Validate and sanitize inputs thoroughly
  • Example:
    • TypeORM automatically parameterizes queries
    • In MongoDB, avoid direct string concatenation for filters

  • Store frequently accessed data to reduce database or API calls
  • In-memory cache (e.g., NestJS CacheModule with default memory store) for simple, fast caching
  • Redis cache for distributed, scalable caching across servers
  • Use @Cacheable(), manual cache management, or interceptors for caching responses
  • Example:
    • @Cacheable()
getCachedData() { /* ... */ }

  • Reduce response payload size to speed up network transfer
  • Use compression middleware (e.g., gzip)
  • Easy integration in NestJS:
    • import * as compression from 'compression';
app.use(compression());

  • Use async factory providers to initialize resources like database connections asynchronously

  • Load modules only when needed instead of at app startup
  • Improves startup time and memory usage
  • Supported via dynamic module imports and imports array in modules
  • Useful for rarely used features or heavy modules

21. Advanced Topics

  • Create reusable decorators to simplify code and add metadata
  • Use createParamDecorator or class/method decorators
  • Example custom param decorator:
    • export const User = createParamDecorator((data, ctx) => {
  const request = ctx.switchToHttp().getRequest();
  return request.user;
});

  • Modules configured and imported with parameters at runtime
  • Return DynamicModule from static forRoot() or forFeature() methods
  • Useful for reusable, configurable libraries or features
  • Example: 
    @Module({})
export class ConfigurableModule {
  static forRoot(options: ConfigOptions): DynamicModule {
    return {
      module: ConfigurableModule,
      providers: [{ provide: 'CONFIG_OPTIONS', useValue: options }],
    };
  }
}

  • Providers instantiated per incoming request (not singleton)
  • Useful when provider holds request-specific state or data
  • Use @Injectable({ scope: Scope.REQUEST })
  • Slightly impacts performance; use only when necessary

  • Combine multiple communication protocols in one NestJS app
  • Import relevant modules (GraphQLModule, WebSocket gateways, REST controllers)
  • Share business logic/services across protocols
  • Example: REST for CRUD, GraphQL for flexible queries, WebSockets for real-time updates

  • Support multiple isolated tenants (clients) within one app instance
  • Strategies:
    • Separate databases per tenant
    • Shared database with tenant ID filtering
  • Implement tenant context injection via middleware or request-scoped providers
  • Ensure data isolation and security

  • Use HTTP modules like HttpModule to call external APIs
  • Handle authentication, rate limiting, error handling for APIs
  • Wrap API calls in services for clean separation
  • Example:

export class ExternalApiService {
