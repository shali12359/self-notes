1. Fundamentals of System Design

Progress:

  • Scalability is the ability of a system to handle increasing load or users without performance degradation.
  • Types of scalability:
    • Vertical scaling (scale-up): Add more resources to a single machine (CPU, RAM, storage).
    • Horizontal scaling (scale-out): Add more machines/servers to share the load.
  • Example: Upgrading a database from 16GB to 64GB RAM is vertical scaling; adding 3 more database servers behind a load balancer is horizontal scaling.

  • Vertical scaling: Pros: Simpler to implement, no distributed system complexity; Cons: Limited by hardware, expensive, single point of failure.
  • Horizontal scaling: Pros: More resilient, can handle very large loads; Cons: More complex to design, needs distributed coordination.

  • Reliability: System performs correctly over time, with minimal failures.
  • Availability: System is accessible and operational when needed.
  • Fault tolerance: System continues to work even if some components fail.
  • Example: A cloud service with multiple redundant servers ensures high availability and fault tolerance.

  • Latency: Time taken to complete a single request (response time).
  • Throughput: Number of requests processed in a given time (e.g., requests/sec).
  • Example: A system may have low latency (fast response) but low throughput if it handles only a few requests at a time.

  • In distributed systems, you can only guarantee two out of three at a time:
    • Consistency: All nodes see the same data at the same time.
    • Availability: Every request gets a response, even if some nodes fail.
    • Partition tolerance: System continues to operate even if network partitions occur.
  • Example: In a network split, you must choose between consistency (return error for some requests) or availability (return potentially stale data).

  • Load balancing: Distributes incoming traffic across multiple servers to avoid overload and improve response time.
    • Types: Round-robin, least connections, IP hash, etc.
    • Example: A web app uses a load balancer to send requests to 5 backend servers.
  • Caching: Stores frequently accessed data closer to the user or app to reduce latency and database load.
    • Types: In-memory caches (Redis, Memcached), CDN caching, browser caching.
    • Example: Cache popular product pages in Redis to serve 10x faster than querying the database every time.

2. Databases & Storage

Progress:

  • Relational (SQL) databases:
    • Structured tables with rows and columns.
    • Use SQL queries for CRUD operations.
    • Strong ACID guarantees.
    • Examples: MySQL, PostgreSQL, Oracle.
  • NoSQL databases:
    • Schema-less or flexible schema, designed for scalability and high performance.
    • Types: Document, Key-Value, Column-family, Graph.
    • Examples: MongoDB (document), Redis (key-value), Cassandra (column-family), Neo4j (graph).
  • Example: Use SQL for banking apps (strong consistency), NoSQL for social media feeds (high scalability).

  • SQL: Structured data, complex queries, transactions.
  • Document (NoSQL): Flexible JSON-like storage, hierarchical data, fast retrieval.
  • Key-Value: Ultra-fast lookup, simple data models, caching.
  • Graph: Relationships-heavy data, social networks, recommendation engines.

  • Replication: Copy data to multiple nodes for high availability and read performance.
  • Sharding (horizontal partitioning): Split data across multiple databases to handle large datasets and writes.
  • Partitioning (vertical/horizontal): Divide tables by columns (vertical) or rows (horizontal) for better performance.
  • Example: User data split across servers based on user ID to distribute load.

  • Indexing: Creates a data structure to speed up query lookups.
    • Example: Indexing email column in users table for fast search.
  • Transactions: Ensure a group of operations either all succeed or all fail.
  • ACID properties:
    • Atomicity: All-or-nothing.
    • Consistency: Database remains in a valid state.
    • Isolation: Concurrent transactions do not interfere.
    • Durability: Committed changes survive crashes.

  • Purpose: Reduce database load and improve performance.
  • Types of caching:
    • LRU cache (Least Recently Used): Evicts the least recently accessed items when full.
    • Write-through cache: Writes go to cache and database simultaneously (ensures consistency).
    • Write-back cache: Writes go first to cache, then asynchronously to database (faster, less DB load).
  • Tools: Redis, Memcached.
  • Example: Store popular product data in Redis to serve millions of users without hitting the database each time.

3. Networking & Communication

Progress:

  • HTTP: Protocol for transferring data between client and server. Stateless, request-response model.
  • HTTPS: HTTP + TLS/SSL encryption for secure communication.
  • Example: Login forms, API requests, and web page loads should use HTTPS to prevent eavesdropping.

  • TCP (Transmission Control Protocol): Reliable, connection-oriented, ensures data arrives in order. Example: Web browsing, file transfer.
  • UDP (User Datagram Protocol): Unreliable, connectionless, faster, low overhead. Example: Video streaming, online gaming.

  • Synchronous APIs: Client waits for server response. Example: REST API call to fetch user data.
  • Asynchronous APIs: Client continues processing; server responds later. Example: Message queues or webhooks.

  • REST: Standard HTTP methods (GET, POST, PUT, DELETE). Easy, widely used, resource-based.
  • GraphQL: Query exactly what you need, single endpoint, reduces over-fetching/under-fetching.
  • gRPC: High-performance, uses Protocol Buffers, supports streaming, ideal for microservices.
  • Example: Use GraphQL for flexible frontend queries; use gRPC for service-to-service communication.

  • Decouple services and enable asynchronous communication.
  • Producers send messages to a queue; consumers process them independently.
  • Supports scaling, retry logic, and loose coupling.
  • Example: Order service pushes “order created” event; inventory service consumes it to update stock.

  • Kafka: High-throughput, distributed log, used for streaming events.
  • RabbitMQ: Reliable message broker, supports routing, queuing, and delivery guarantees.
  • SQS (AWS): Fully managed queue service, scalable, reliable for cloud apps.

  • Full-duplex communication between client and server over a single TCP connection.
  • Enables real-time updates without repeated HTTP requests.
  • Example: Chat apps, live notifications, multiplayer games.

4. System Components & Architecture

Progress:

  • Load balancers: Distribute incoming traffic across multiple servers to prevent overload and improve reliability.
  • Types: Round-robin, least connections, IP hash.
  • Example: A web app with 5 backend servers uses a load balancer to ensure no single server is overwhelmed.
  • Reverse proxies: Sit between clients and servers, handling requests on behalf of servers. Often used for caching, SSL termination, and security.
  • Example: Nginx or HAProxy acting as a reverse proxy for a web application.

  • Networks of servers located globally to deliver static content (images, videos, scripts) quickly.
  • Reduce latency and server load by caching content closer to users.
  • Example: A video streaming service uses CloudFront or Akamai to serve videos faster to users worldwide.

  • Monolith: Single, unified codebase. Simple initially, harder to scale or maintain over time.
  • Microservices: Application split into independent services communicating via APIs. Scalable, maintainable, but adds complexity.
  • Serverless: Functions run in response to events; no server management. Great for variable workloads, but limited control and cold-start latency.
  • Example: E-commerce platform: microservices for payments, orders, and inventory; serverless for email notifications.

  • Tasks executed in the background, without blocking the main request flow.
  • Useful for time-consuming tasks like sending emails, generating reports, or processing videos.
  • Example: User uploads a video; encoding happens asynchronously while the user continues browsing.

  • Worker queues hold tasks to be processed asynchronously by background workers.
  • Ensures reliability and scalability by decoupling task execution from the main application flow.
  • Tools: RabbitMQ, Kafka, Celery, SQS.
  • Example: Order service pushes a “send receipt email” task to a queue; workers process it independently.

  • Rate limiting: Limits number of requests a user or client can make in a given time. Prevents abuse.
  • Throttling: Controls the traffic sent to a service to avoid overload.
  • Circuit breakers: Detect failing services and stop sending requests temporarily to avoid cascading failures.
  • Example: An API allows 100 requests/minute per user; if a downstream service is failing, the circuit breaker prevents further requests until it recovers.

5. Design Patterns & Principles

Progress:

  • Common architectural patterns help structure systems for scalability, maintainability, and reliability.
  • Examples:
    • Client-server: Central server serves multiple clients.
    • Peer-to-peer (P2P): Each node can act as client and server.
    • Publish-subscribe (pub-sub): Publishers send messages to channels; subscribers receive updates asynchronously.
    • Event sourcing: All state changes are stored as a sequence of events, allowing reconstruction of state at any point.
  • Example: Chat apps often use pub-sub to broadcast messages to multiple users.

  • SOLID principles: Guidelines for writing clean, maintainable, and flexible object-oriented code:
    • S: Single Responsibility Principle (one class = one purpose)
    • O: Open/Closed Principle (open for extension, closed for modification)
    • L: Liskov Substitution Principle (subtypes can replace base types without issues)
    • I: Interface Segregation Principle (prefer many small interfaces over a large one)
    • D: Dependency Inversion Principle (depend on abstractions, not concrete classes)
  • Modular design: Break the system into independent, reusable modules for easier development and testing.
  • Example: Separate payment processing, notifications, and user management into independent modules.

  • Idempotent operations produce the same result even if repeated.
  • Important in distributed systems where network retries can cause duplicate requests.
  • Example: Submitting an order payment multiple times due to network failure should charge the customer only once.

  • Used to handle temporary failures in distributed systems.
  • Types:
    • Simple retry: Try again after failure.
    • Exponential backoff: Gradually increase delay between retries to reduce load.
    • Jitter: Add randomness to prevent thundering herd problem.
  • Example: If a service call fails due to network issues, retry after 1s, 2s, 4s… with small random jitter.

6. Scalability & Performance

Progress:

  • Adding more servers/nodes to distribute load across multiple machines.
  • Improves system capacity and reliability.
  • Example: A web application adds 3 more backend servers behind a load balancer to handle increased traffic.

  • Partitioning (Sharding): Divide data across multiple databases or servers to reduce load per node.
    • Horizontal partitioning splits by rows (e.g., user ID ranges).
    • Vertical partitioning splits by columns (e.g., user info vs analytics).
  • Replication: Copy data to multiple nodes to improve read performance and provide high availability.
  • Example: User data is sharded by user ID across multiple database servers, with each shard replicated twice for redundancy.

  • Temporarily store frequently accessed data closer to the application to reduce latency and database load.
  • Types of caching:
    • In-memory: Redis, Memcached
    • CDN caching: For static content like images, scripts, and videos
  • Example: Cache product catalog in Redis to serve thousands of requests per second without hitting the database.

  • Reuse a pool of database connections instead of creating a new connection per request.
  • Reduces latency and resource overhead for high-traffic applications.
  • Example: Web server maintains 50 open connections to a database, reusing them for incoming requests.

  • Content Delivery Network (CDN) caches static assets (images, CSS, JS) on servers distributed globally.
  • Reduces latency, improves load times, and decreases origin server load.
  • Example: A website uses CloudFront or Akamai to deliver images and videos faster to users worldwide.

7. Reliability & Monitoring

Progress:

  • Health checks: Regularly check if servers, services, or databases are operational.
  • Can be simple HTTP endpoint checks or custom scripts.
  • Failover strategies: Automatically switch traffic to backup systems if the primary system fails.
  • Improves system availability and fault tolerance.
  • Example: Load balancer detects a downed server via health check and reroutes requests to healthy servers.

  • Logging: Record events, errors, and system activity for debugging and auditing.
  • Best practice: Include timestamps, request IDs, and context.
  • Monitoring: Track system health, performance, and resource usage in real-time.
  • Prometheus: Collects and stores metrics from applications/services.
  • Grafana: Visualizes metrics in dashboards for easy analysis.
  • Example: Track CPU/memory usage of all servers and display trends on a Grafana dashboard.

  • Set up alerts for abnormal behavior (high latency, failed requests, low disk space).
  • Integrate with incident response workflows to quickly investigate and resolve issues.
  • Example: PagerDuty triggers an alert if API response time exceeds 2 seconds for 5 consecutive minutes.

  • Latency: Time to respond to a request.
  • Throughput: Number of requests processed per unit time.
  • Error rate: Percentage of failed requests.
  • Resource utilization: CPU, memory, disk, and network usage.
  • Example: Monitoring error rates helps identify failing services before users notice.

8. Security Considerations in Design

Progress:

  • Ensure users and services are properly identified (authentication) and can only access permitted resources (authorization).
  • Use centralized identity providers (OAuth2, OpenID Connect, JWT) for consistent and scalable security.
  • Example: A multi-service platform uses a single identity server to issue tokens, allowing all services to verify user access without separate login systems.

  • Protect APIs from misuse and attacks:
  • Validate input and output to prevent SQL injection or XSS.
  • Use HTTPS for secure communication.
  • Implement proper authentication and authorization.
  • Example: A payment API requires an access token for each request and validates the request body before processing.

  • Rate limiting: Restrict number of requests per client to prevent abuse.
  • DDoS protection: Use load balancers, WAFs, and CDNs to absorb and mitigate traffic spikes.
  • Example: API allows 100 requests/minute per user; traffic beyond this is throttled to protect the system.

  • In transit: Encrypt data sent between clients and servers using TLS/HTTPS.
  • At rest: Encrypt stored data in databases or storage systems to prevent exposure if compromised.
  • Example: Customer payment info is encrypted in the database and all API requests use HTTPS to prevent interception.

9. Real-world Design Case Studies

Progress:

  • Objective: Convert long URLs into short, unique links.
  • Considerations:
    • Generate unique keys (hashing, base62 encoding).
    • Store mappings efficiently in a database (key-value store like Redis or DynamoDB).
    • Handle high read traffic using caching.
  • Scaling strategies: Partition data by key ranges, replicate database for reliability.
  • Example: bit.ly serves billions of short links with minimal latency.

  • Objective: Show users a personalized, timely feed.
  • Considerations:
    • Feed generation: push (precompute) vs pull (compute on request).
    • Store user posts, likes, comments efficiently.
    • Handle millions of concurrent users and updates.
  • Scaling strategies: Caching popular feeds, sharding user data, using message queues for asynchronous feed updates.
  • Example: Twitter uses distributed storage and caching to serve feeds quickly.

  • Objective: Real-time messaging between users.
  • Considerations:
    • Low-latency delivery using WebSockets.
    • Message persistence in database or distributed log.
    • Presence and typing indicators.
  • Scaling strategies: Partition users across multiple servers, use pub-sub or message brokers (Kafka, RabbitMQ) for message delivery.
  • Example: WhatsApp uses Erlang servers for scalable, real-time messaging.

  • Objective: Deliver video content efficiently to users.
  • Considerations:
    • Store and encode videos in multiple resolutions.
    • Use CDNs to reduce latency.
    • Adaptive bitrate streaming to adjust quality per user bandwidth.
  • Scaling strategies: Cache popular videos at edge servers, partition storage and replicate content.
  • Example: YouTube serves billions of videos using distributed storage and global CDNs.

  • Objective: Match riders with nearby drivers in real-time.
  • Considerations:
    • Real-time location tracking and geospatial queries.
    • Dynamic pricing and surge calculation.
    • High concurrency during peak hours.
  • Scaling strategies: Use sharded databases for trip and driver data, employ in-memory caches for nearby driver lookups, asynchronous queues for notifications and payment processing.
  • Example: Uber uses distributed services, caching, and real-time geolocation to handle millions of rides.

  • Identify hotspots: database writes, API endpoints, or caching layers.
  • Optimize data flow: asynchronous processing, message queues, caching.
  • Plan scaling strategies: vertical/horizontal scaling, sharding, replication, CDNs.
  • Monitor metrics: latency, throughput, error rates, and adjust system design accordingly.
