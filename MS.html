1. Fundamentals of Microservices

Progress:

  • An architectural style where an application is built as a collection of small, independent services. Each service runs in its own process and communicates via lightweight protocols (HTTP/REST, gRPC, or messaging).
  • Characteristics:
    • Independent deployability → each service can be built, tested, and deployed without affecting others.
    • Decentralized data management → each service owns its own database/schema.
    • Technology flexibility → different services can use different programming languages or frameworks.
    • Resilience → failure in one service should not bring down the entire system.
    • Scalability → services can scale independently based on demand.

  • Monolithic:
    • Single codebase, all modules tightly coupled.
    • Easier to develop initially but harder to scale and maintain as the app grows.
    • Deployment → the entire app must be redeployed even for small changes.
  • Microservices:
    • Application split into small, autonomous services.
    • Each service can scale independently.
    • Easier to maintain and evolve large applications.
    • Requires more infrastructure (service discovery, communication, monitoring).

  • Benefits:
    • Faster development & deployment cycles.
    • Independent scaling → saves resources.
    • Fault isolation → one service failing won’t crash the whole system.
    • Technology diversity → best tool for each service.
  • Challenges:
    • Complexity → managing multiple services is harder than one monolith.
    • Data consistency → hard to maintain across distributed databases.
    • Operational overhead → need DevOps, CI/CD, monitoring, logging.
    • Network latency → services communicate over the network, which adds overhead.

  • Single Responsibility: Each microservice should do one thing really well (e.g., "User Service" only manages users, not payments).
  • Loose Coupling: Services should minimize dependencies. One service’s change should not break others. Communication via well-defined APIs.
  • High Cohesion: All logic related to a specific business capability should be inside one service (e.g., all “order-related” logic lives in the Order Service).

  • Use Cases (when to use):
    • Large, complex applications with multiple business domains.
    • Applications needing independent scaling (e.g., streaming service where “video streaming” needs more scaling than “user profiles”).
    • Teams working on different features simultaneously (microservices allow parallel development).
  • When NOT to use:
    • Small/simple applications where a monolith is faster and easier to maintain.
    • Teams without enough DevOps maturity (microservices require advanced CI/CD, monitoring, and deployment practices).
    • When low latency is critical and network calls could be a bottleneck.

2. Microservices Design Patterns

Progress:

  • A single entry point for all client requests, which routes them to the correct microservice.
  • Hides service complexity, provides security, handles load balancing, request aggregation, and authentication.
  • Example: Instead of calling 5 different services separately, the client calls the API Gateway, which aggregates and returns the response.

  • Mechanism that helps microservices automatically find each other at runtime.
  • Service Registry: A database of available services and their network locations (e.g., Eureka, Consul).
  • Service Discovery: Allows services to dynamically register and discover other services without hardcoding addresses.
  • Makes services scalable and fault-tolerant in dynamic environments like Kubernetes.

  • Each microservice has its own database, instead of sharing a single one.
  • Ensures loose coupling and data ownership per service.
  • Challenge: Maintaining data consistency across services (needs eventual consistency).

  • Separates read (queries) and write (commands) operations into different models.
  • Improves scalability, performance, and allows optimized read/write handling.
  • Example: Writing to one DB while reading from a read-optimized DB (e.g., caching or projections).

  • Manages transactions across multiple services using a sequence of local transactions with compensation steps.
  • Types:
    • Choreography: Services listen to events and react.
    • Orchestration: A central controller coordinates steps.
  • Ensures data consistency without two-phase commit (which doesn’t scale in microservices).

  • Instead of storing just the current state in the database, store a log of all state-changing events.
  • Provides an audit trail, allows replaying events to rebuild state, and works well with CQRS.
  • Challenge: Event log management and complexity of replaying.

  • A strategy for migrating from a monolith to microservices gradually.
  • New features are built as microservices, while old parts remain in the monolith until replaced.
  • Reduces risk by avoiding a "big bang" migration.

3. Service Communication

Progress:

  • REST:
    • REST (Representational State Transfer) is an architectural style using HTTP for communication between services.
    • Characteristics:
      • Human-readable (JSON/XML).
      • Stateless – each request contains all necessary information.
      • Widely adopted, easy to debug.
    • Use Cases: Best when simplicity and human readability are more important than performance.
  • gRPC:
    • gRPC is a high-performance, open-source RPC framework that uses Protocol Buffers (Protobuf) for serialization.
    • Characteristics:
      • Faster and more efficient than REST (binary format).
      • Supports streaming (client, server, bi-directional).
      • Strongly typed contracts through Protobuf definitions.
    • Use Cases: Ideal for low-latency, high-throughput internal service communication.

  • Message Brokers (RabbitMQ, Kafka, NATS):
    • Middleware systems that allow services to communicate by sending messages via queues or topics.
    • Characteristics:
      • Decouples sender and receiver.
      • Ensures durability, reliability, and scalability.
      • Kafka → high throughput, event streaming.
      • RabbitMQ → strong in complex routing and traditional queuing.
      • NATS → lightweight, simple, low-latency pub/sub.
    • Use Cases: When services need to work independently without waiting for responses (e.g., order processing, notifications).
  • Event-Driven Architecture:
    • A style where services produce and consume events to react to changes in state.
    • Characteristics:
      • Promotes real-time processing.
      • Loose coupling between services.
      • Facilitates scalability and resilience.
    • Use Cases: Best for systems where business processes depend on state changes (e.g., stock updates, payment confirmations).

  • Request/Response:
    • Communication style where one service sends a request and waits for a response.
    • Common in synchronous protocols (REST, gRPC).
    • Tight coupling → sender expects a reply from a specific service.
    • Example: A user service querying an authentication service.
  • Publish/Subscribe:
    • One service publishes messages/events to a topic; multiple services can subscribe and react.
    • Common in asynchronous/event-driven architectures.
    • Loose coupling → publishers don’t need to know who consumes the event.
    • Example: An order service publishing an “OrderPlaced” event consumed by inventory, billing, and notification services.

4. Data Management in Microservices

Progress:

  • Each microservice owns its database and manages its schema independently.
  • This ensures loose coupling — changes in one service’s data model don’t affect others.
  • Example: The Orders service has its own DB, while the Payments service has a separate one.

  • If multiple services share the same database, it creates tight coupling.
  • Schema changes by one service can break others.
  • It becomes harder to enforce service boundaries and ownership of data.
  • Leads to coordination overhead and reduced autonomy of teams.

  • Different services can use different types of databases depending on their needs.
  • Example:
    • Catalog service → NoSQL database for flexible schema.
    • Orders service → Relational database for strong consistency.
  • Enables using the right tool for the job instead of forcing a single DB for all services.

  • Since each service has its own database, sometimes data needs to be replicated across services.
  • Approaches:
    • Event-driven replication → services publish events (e.g., “OrderCreated”) and others subscribe to update their local DB.
    • API calls for sync → one service calls another to fetch/update data.
  • Challenges: avoiding data staleness and ensuring consistency.

  • Traditional ACID transactions across multiple services/databases are hard to implement.
  • Instead, microservices rely on eventual consistency: data will become consistent over time.
  • Strategies:
    • Saga pattern → break transactions into smaller steps with compensating actions if one fails.
    • Event sourcing → store events as the source of truth and rebuild state as needed.
  • Trade-off: lose strong consistency, but gain scalability and resilience.

5. Service Discovery

Progress:

  • In a microservices architecture, services often run on multiple instances that may scale up or down, and their network locations (IP addresses, ports) change dynamically. Service Discovery is the process of automatically detecting and keeping track of these service instances so that they can communicate with each other without hardcoding addresses.

  • The client is responsible for finding available service instances.
  • Client queries the service registry (a database of active services).
  • After getting the list of instances, the client selects one (e.g., using round-robin) and sends the request directly.
  • Example: Netflix Eureka with Ribbon (client load balancer).
  • Steps: Service registers → Client queries registry → Client calls service.
  • Pros: Simple, no middleman needed.
  • Cons: Client must handle discovery logic and load balancing, making it more complex.

  • The client sends a request to a load balancer or API Gateway instead of looking up services itself.
  • The load balancer queries the service registry and picks a healthy instance.
  • The client doesn’t need to know about the discovery process.
  • Example: AWS Elastic Load Balancer.
  • Steps: Service registers → Client sends request to load balancer → Load balancer queries registry → Request routed to service.
  • Pros: Client is simpler, centralizes load balancing.
  • Cons: Adds an extra network hop and dependency on the load balancer.

  • Netflix Eureka:
    • Popular with Spring Cloud.
    • Provides service registry and supports client-side discovery.
    • Lightweight and integrates well with Ribbon and Zuul.
  • Consul:
    • Developed by HashiCorp.
    • Provides service registry, health checks, and key-value storage.
    • Supports both client-side and server-side discovery.
    • Often used in containerized environments.
  • Zookeeper:
    • Originally from Apache Hadoop ecosystem.
    • Distributed coordination service, often used for leader election, configuration, and service discovery.
    • Strong consistency but heavier compared to Eureka and Consul.

6. API Gateway

Progress:

  • An API Gateway acts as a single entry point for all clients to access multiple microservices. It handles request routing, security, and other cross-cutting concerns, simplifying client interaction with a microservices architecture.

  • Centralizes client requests and routes them to appropriate microservices.
  • Benefits:
    • Simplifies client interactions (one endpoint instead of many).
    • Handles cross-cutting concerns like authentication, logging, and monitoring.
    • Aggregates responses from multiple services.
    • Enables versioning and backward compatibility.

  • API Gateway can enforce security policies before requests reach microservices.
  • Supports:
    • Token-based authentication (JWT, OAuth2).
    • Role-based access control (RBAC).
    • Integration with identity providers.
  • Ensures that only authorized clients can access protected services.

  • Routing: Directs requests to the appropriate microservice based on URL path, headers, or request parameters.
  • Load balancing: Distributes incoming requests across multiple instances of a service for better performance and fault tolerance.

  • Rate limiting: Controls the number of requests a client can make in a given time, preventing abuse or overload.
  • Caching: Stores responses temporarily to reduce service load and improve performance.
  • These features improve reliability, scalability, and response times.

  • Kong: Cloud-native API gateway with plugins for authentication, logging, and rate limiting.
  • Nginx: Web server often used as reverse proxy, load balancer, and API gateway.
  • AWS API Gateway: Fully managed gateway for creating, publishing, and monitoring APIs on AWS.
  • Zuul: Netflix open-source gateway, integrates well with Spring Cloud for routing and filtering.

7. Security in Microservices

Progress:

  • Microservices communicate over networks, often exposed to the internet, making security critical. Key areas include authentication, authorization, secure communication, and secret management.

  • Authentication: Verifies the identity of a user or service.
  • Authorization: Determines what resources or actions the authenticated user/service can access.
  • Protocols & Standards:
    • OAuth 2.0:
      • Delegated authorization framework.
      • Allows clients to access resources on behalf of users without sharing credentials.
      • Common flows: Authorization Code, Client Credentials, Implicit, Resource Owner Password.
    • OpenID Connect (OIDC):
      • Built on top of OAuth 2.0.
      • Adds user identity verification and profile information.
      • Provides ID tokens for authentication in addition to access tokens.
    • JWT (JSON Web Token):
      • Compact, self-contained token used for authentication/authorization.
      • Includes claims (user info, roles, expiry) and is signed to ensure integrity.
      • Commonly used in stateless authentication between services.

  • Validate inputs to prevent injection attacks.
  • Use HTTPS to encrypt communication.
  • Rate limit requests to prevent abuse.
  • Implement logging and monitoring for suspicious activity.
  • Avoid exposing internal endpoints to public networks.

  • mTLS ensures two-way authentication between client and server.
  • Both parties present certificates to verify identities.
  • Enhances security by encrypting traffic and confirming service authenticity.
  • Commonly used for service-to-service communication in microservices.

  • Microservices often need credentials (DB passwords, API keys, tokens).
  • Secrets should not be hard-coded; use secure secret management tools.
  • Examples:
    • HashiCorp Vault → Centralized secret storage with dynamic secrets and access policies.
    • AWS Secrets Manager → Managed service to store, rotate, and retrieve secrets securely.

8. Deployment & Orchestration

Progress:

  • Microservices require careful deployment and orchestration because of their distributed nature. Key practices include containerization, orchestration, and advanced deployment strategies.

  • Packaging microservices and their dependencies into lightweight, portable containers.
  • Benefits:
    • Consistent environment across development, testing, and production.
    • Simplifies deployment and scaling.
    • Isolates services to avoid conflicts.
  • Example: `docker build` to create an image and `docker run` to start a container.

  • Automates deployment, scaling, networking, and management of multiple containers.
  • Popular Tools:
    • Kubernetes:
      • Open-source orchestration platform.
      • Manages pods, deployments, services, scaling, and self-healing.
      • Handles service discovery and load balancing automatically.
    • Docker Swarm:
      • Native Docker orchestration.
      • Easier to set up than Kubernetes but less feature-rich.
      • Manages container clusters and simple scaling.

  • Infrastructure layer for managing service-to-service communication.
  • Features:
    • Traffic routing, retries, and failover.
    • Observability and metrics.
    • Security (mTLS between services).
  • Example Tools: Istio, Linkerd.

  • Rolling Updates: Gradually replace old service instances with new ones without downtime.
  • Blue-Green Deployment: Maintain two identical environments (Blue = current, Green = new). Switch traffic to the Green environment once it’s validated.
  • Benefits: Zero-downtime deployment, easy rollback, and minimal disruption.

9. Monitoring & Observability

Progress:

  • Monitoring and observability are critical in microservices because distributed services can fail independently, making it challenging to detect and debug issues.

  • Aggregating logs from all microservices into a single system for easier analysis.
  • ELK Stack:
    • Elasticsearch → Stores and indexes log data for fast searches.
    • Logstash → Collects, processes, and forwards logs.
    • Kibana → Visualizes logs with dashboards and search queries.
  • Benefits:
    • Easy debugging of issues across multiple services.
    • Detects patterns and anomalies.

  • Tracks requests as they propagate through multiple services, helping identify latency bottlenecks.
  • Tools:
    • Jaeger → Open-source distributed tracing with visualization and analysis.
    • Zipkin → Collects and visualizes traces; integrates with Spring Cloud Sleuth.
  • Benefits:
    • Helps understand request flows across services.
    • Pinpoints slow services or failed calls.

  • Quantitative measurements of system behavior over time.
  • Tools:
    • Prometheus → Collects metrics from services, stores time-series data, and triggers alerts.
    • Grafana → Visualizes metrics with dashboards, graphs, and alerts.
  • Benefits:
    • Monitors performance, resource usage, and system health.
    • Helps in capacity planning and scaling decisions.

  • Checks whether services are alive and responsive.
  • Implementation:
    • Spring Boot’s /actuator/health endpoint.
    • External uptime monitoring tools.
  • Benefits:
    • Automatic detection of failed services.
    • Supports automated failover and alerts.

10. Resilience & Fault Tolerance

Progress:

  • Microservices are distributed, so failures in one service can cascade and affect others. Resilience and fault tolerance ensure that the system continues to function gracefully under failure conditions.

  • Prevents repeated calls to a failing service to avoid cascading failures.
  • How it works:
    • Monitors service calls; if failures exceed a threshold, the circuit “opens” and subsequent calls fail immediately.
    • After a timeout, the circuit “half-opens” to test if the service has recovered.
  • Tools: Netflix Hystrix, Resilience4j.
  • Benefit: Improves system stability and prevents overload on failing services.

  • Automatically retries failed service calls to handle transient errors.
  • Exponential Backoff: Wait time increases after each retry (e.g., 1s → 2s → 4s) to reduce system load.
  • Benefit: Helps recover from temporary network or service glitches without overwhelming services.

  • Isolates resources (threads, connections) for each service or component to prevent failures from affecting others.
  • Analogy: Like compartments in a ship; if one leaks, the rest remain intact.
  • Benefit: Prevents cascading failures and improves system stability.

  • Automatically switch to a backup service or instance when the primary fails.
  • Examples:
    • Active-passive failover → standby instance is idle until primary fails.
    • Active-active failover → multiple instances handle traffic simultaneously; traffic shifts on failure.
  • Benefit: Ensures high availability and continuity of service.

11. Testing Microservices

Progress:

  • Focuses on testing small, isolated pieces of code (functions, classes, methods).
  • Ensures microservice logic works as expected.
  • Frameworks: JUnit (Java), pytest (Python), Jest (JavaScript).
  • Benefits: Fast execution, immediate developer feedback.

  • Validates how different modules or services work together.
  • Example: Checking whether a service properly stores and retrieves data from a database.
  • Ensures correct communication via APIs, databases, or messaging systems.
  • Slower than unit tests but essential for detecting interface issues.

  • Ensures services agree on request and response structures.
  • Use Case: Useful when multiple teams independently build services that must communicate.
  • Example: Consumer (frontend) defines expected request/response; Provider (backend) ensures compliance.
  • Tool: Pact for consumer-driven contract testing.

  • Validates entire system flows across multiple microservices.
  • Example: Simulating a user placing an order in an e-commerce app.
  • Ensures frontend, backend, and services work together correctly.
  • Slowest type of test but crucial for verifying business workflows.

  • Used when external systems (databases, APIs, payment gateways) are unavailable or costly during testing.
  • Simulates behavior of real dependencies to isolate tests.
  • Benefit: Speeds up testing, avoids dependency costs.
  • Example: Mock a payment API to always return success.

12. Scalability & Performance

Progress:

  • Vertical Scaling (Scaling Up):
    • Adding more resources (CPU, RAM, storage) to a single server.
    • Example: Upgrading a 4-core CPU server to 16 cores.
    • Pros: Simple to implement, no app changes required.
    • Cons: Hardware limit, single point of failure, costly.
  • Horizontal Scaling (Scaling Out):
    • Adding more servers/machines to distribute the load.
    • Example: Adding 10 smaller servers instead of one large server.
    • Pros: Better fault tolerance, virtually unlimited scaling, cost-effective with cloud.
    • Cons: Requires load balancing, data distribution, more complex architecture.
  • In microservices, horizontal scaling is preferred since services are stateless and easy to replicate.

  • Ensures traffic is evenly distributed across multiple service instances.
  • Strategies:
    • Round Robin: Requests distributed sequentially. Simple and fair.
    • Least Connections: Sends requests to the server with the fewest active connections. Best for uneven workloads.
    • IP Hash / Session Affinity: Routes client requests to the same server. Useful when sessions are stateful (though stateless is preferred).
    • Weighted Load Balancing: Assigns more requests to higher-capacity servers.
  • Load balancers also improve availability and handle failover.

  • Reduces latency and offloads databases.
  • Types of Caching:
    • Client-side: Stored in browser/local device.
    • Server-side: Cached in API gateway or application.
    • Database caching: Stores query results for faster responses.
  • Tools:
    • Redis: In-memory caching with persistence, pub/sub, advanced data structures.
    • Memcached: Simple, fast key-value caching, no persistence.
  • Caching reduces DB load, speeds response, but cache invalidation is tricky.

  • Validates system behavior under load.
  • JMeter:
    • GUI-based and scriptable.
    • Supports load testing for web apps, APIs, DBs.
    • Good for functional + performance testing.
  • Gatling:
    • Code-based tool (Scala DSL).
    • High performance, CI/CD friendly.
    • Efficient for large-scale load tests.
  • Performance testing ensures scalability, stability, and reliability at peak workloads.

13. Event-Driven Microservices

Progress:

  • Instead of storing only the latest state, every change (event) is stored as a sequence.
  • Example: In a bank account, all deposits and withdrawals are recorded as events instead of just storing the balance.
  • Benefits:
    • Full audit trail (history of all changes).
    • Easy to rebuild state by replaying events.
    • Supports temporal queries (e.g., “Balance last week”).

  • Message Broker: Middleware system that handles communication between microservices via messages/events.
  • Apache Kafka:
    • Distributed streaming platform.
    • Best for high-throughput event streaming.
    • Stores events in a durable, scalable log.
  • RabbitMQ:
    • Traditional message broker.
    • Reliable message delivery, routing, and queuing.
    • Often used for task queues and async communication.
  • Why Use Them?: Decouples services, improves scalability, and enables async processing.

  • Event Notification: Lightweight events notifying other services that something happened.
  • Event-Carried State Transfer: Events contain full state, so consumers don’t need to query back.
  • Event Sourcing: Events are the source of truth, state rebuilt by replaying events.
  • CQRS (Command Query Responsibility Segregation):
    • Commands → change state (via events).
    • Queries → read from optimized read models.

  • Schema Design: Define how event data is structured (fields, types, metadata). 
    {
  "event": "UserRegisteredEvent",
  "userId": "123",
  "email": "test@example.com",
  "registeredAt": "2024-08-17T10:00:00Z"
}
  • Versioning:
    • Add optional fields (don’t break old readers).
    • Never remove required fields.
    • Use schema registries (e.g., Confluent Schema Registry for Kafka).

14. CI/CD in Microservices

Progress:

  • Git as the single source of truth for managing infra & apps.
  • Store configs (YAMLs, Helm charts, Terraform) in Git → changes merged → automation tools (ArgoCD, Flux) sync with cluster.
  • Benefits:
    • Clear audit history.
    • Easy rollback.
    • Automation & consistency across environments.

  • CI (Continuous Integration): Runs unit tests, builds Docker images, scans code quality/security.
  • CD (Continuous Delivery/Deployment): Deploys new version to staging/production.
  • Pipelines must handle:
    • Independent deployments.
    • Dependency management (shared APIs/contracts).
    • Automation (test → build → deploy).
  • Tools: Jenkins, GitHub Actions, GitLab CI, CircleCI.

  • Release strategy where a small % of traffic (e.g., 5%) is routed to new version first.
  • Gradually increase traffic until 100% if no errors.
  • Tools: Kubernetes, Istio, Argo Rollouts.
  • Benefits: Reduces risk, enables quick rollback if issues arise.

  • Manage infrastructure (servers, networks, DBs) using code.
  • Terraform:
    • Declarative (“what infra should look like”).
    • Supports multi-cloud (AWS, GCP, Azure).
  • Ansible:
    • Procedural/config management (“how to configure”).
    • Used for automating server setup & deployments.
  • Benefits: Consistency, scalability, repeatability, easier disaster recovery.

15. Domain-Driven Design (DDD)

Progress:

  • DDD is an approach to software design that focuses on modeling the software based on the business domain. The goal is to ensure that the system structure and language align closely with the business needs and rules.

  • Defines clear boundaries around a domain model. Terms & concepts have specific meanings within each context.
  • Example: In e-commerce:
    • Order Context → placing & tracking orders.
    • Payment Context → billing & transactions.
  • Contexts interact via APIs or messaging.
  • Ensures clarity & prevents mixing business rules.

  • Entities: Have unique identity (e.g., Customer with ID).
  • Value Objects: Defined by attributes, no ID (e.g., Address).
  • Aggregates: Cluster of entities & value objects as one unit.
    • Aggregate root enforces consistency.
    • Example: Order aggregate → contains OrderItems.
  • Prevents inconsistent updates & keeps model robust.

  • Translator between bounded contexts with different models/terminologies.
  • Example: Payment expects amountInCents, Order uses amountInDollars → ACL handles conversion.
  • Keeps services independent & prevents "corruption" of domain logic.

  • Strategic Design: High-level structure → focuses on bounded contexts & relationships (e.g., Order, Inventory, Payment contexts).
  • Tactical Design: Detailed design → entities, aggregates, value objects, repositories, services.
  • Example: Order aggregate validation belongs in tactical design.
  • Strategic = big picture; Tactical = implementation details.

16. Governance & Best Practices

Progress:

  • APIs evolve, and backward-incompatible changes can break clients. Versioning ensures smooth transitions.
  • Approaches:
    • URI versioning → /api/v1/users
    • Header versioning → Accept: application/vnd.myapp.v1+json
    • Query parameter versioning → /api/users?version=1
  • Best practice: Communicate deprecation timelines & keep multiple versions only as needed.

  • Helps developers & teams understand how to use a service, reducing miscommunication.
  • Key points:
    • Industry standard for describing REST APIs.
    • Provides interactive docs (Swagger UI).
    • Enables contract-first development.
    • Generates client SDKs automatically.
  • Best practice: Keep docs in sync with code (prefer auto-generation).

  • Poor dependency management causes version conflicts, security issues, & bloated builds.
  • Best practices:
    • Use dependency lock files (e.g., package-lock.json, pom.xml).
    • Update dependencies regularly to patch vulnerabilities.
    • Avoid tight coupling → define clear contracts (API, events).

  • Each service must have a clear owner to avoid “no one’s responsible.”
  • Best practices:
    • Apply Conway’s Law → service boundaries reflect team boundaries.
    • Teams own services end-to-end (code → production).
    • Define SLAs (agreements) & SLOs (objectives).
    • Encourage autonomy while enforcing shared governance (security, logging, monitoring).

17. Migration Strategies

Progress:

  • A monolith is a single, tightly coupled application where all modules share the same codebase & database.
  • How migration works:
    • Identify modules (e.g., user management, billing, catalog).
    • Extract modules into independent microservices with their own APIs & data stores.
    • Use the Strangler Fig Pattern → gradually replace monolith parts until fully phased out.

  • Avoids risky “big bang” rewrites by migrating step by step.
  • Approaches:
    • API façade → Monolith stays intact, new features come from microservices.
    • Data ownership shift → Move data responsibility to microservices in chunks.
    • Feature toggles → Route selected features to microservices while others stay in monolith.
  • Benefit: Lower risk, continuous delivery, and stable transitions.

  • Monolith & microservices coexist during migration.
  • Examples:
    • Monolith handles core functions, new features go into microservices.
    • Gateway routes requests between monolith & microservices.
    • Temporary shared databases may exist, but goal = service owns its own data.
  • Benefit: Ensures business continuity during transition.
