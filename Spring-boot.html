1. Spring Boot Fundamentals

  • Spring Framework: A popular Java framework for enterprise apps.
    • Features: Dependency Injection (IoC), Transaction Management, AOP, integrations.
  • Spring Boot: Lightweight extension of Spring Framework.
    • Reduces boilerplate and simplifies configuration.
    • Quick setup for production-ready apps.

  • Spring:
    • Manual configuration (XML/Java).
    • Flexible but more setup required.
    • No built-in server (Tomcat/Jetty required).
  • Spring Boot:
    • Auto-configuration support.
    • Embedded servers (Tomcat, Jetty, Undertow).
    • Opinionated defaults for faster dev.
    • spring-boot-starter dependencies simplify setup.

  • Auto-Configuration: Configures beans based on classpath/settings.
  • Embedded Server: Standalone JAR (Tomcat/Jetty).
  • Starter Dependencies: Bundled dependencies (e.g., spring-boot-starter-web).
  • Production Ready: Health checks, metrics, monitoring (Actuator).
  • Rapid Development: Minimal setup, quick prototype/deploy.

  • Maven:
    • XML-based build tool (pom.xml).
    • Strong dependency management.
    • Uses central repository.
  • Gradle:
    • Groovy/Kotlin DSL-based (build.gradle).
    • Faster with incremental builds.
    • Preferred for larger projects.
  • Spring Initializr: Ready-to-use templates for Maven & Gradle.

2. Spring Core Concepts

  • IoC: The container creates, configures, and manages objects instead of the programmer.
  • Reads metadata (annotations, XML, or Java config) and injects dependencies into beans.
  • Types of IoC containers:
    • BeanFactory: Basic container, lazy loading.
    • ApplicationContext: Advanced container, supports i18n, events, eager loading.

  • @Component: Generic stereotype for any Spring-managed bean.
  • @Service: Specialized @Component for service layer beans (business logic).
  • @Repository: Specialized @Component for DAO layer beans.
    • Adds persistence-related exception translation.
  • All three make a class eligible for component scanning so Spring can detect/manage it as a bean.

  • @Autowired: Instructs Spring to inject a bean dependency (constructor, setter, or field).
  • Constructor Injection (recommended): 
    @Service
public class UserService {
    private final UserRepository repo;
                
    @Autowired
    public UserService(UserRepository repo) {
        this.repo = repo;
    }
}
  • Benefits:
    • Makes dependencies explicit.
    • Easier unit testing.
    • Ensures immutability (final dependencies).

  • @Bean: Used inside a @Configuration class to explicitly define a bean.
  • Example: 
    @Configuration
public class AppConfig {
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
  • Use case: When you need more control over bean creation than component scanning provides.

  • singleton (default): One shared instance per container.
  • prototype: New instance every time it’s requested.
  • request: One instance per HTTP request (web apps).
  • session: One instance per HTTP session.
  • application: One instance per ServletContext lifecycle.
  • ➡️ Scope defines bean lifecycle & visibility.

  • By default, Spring creates singleton beans at startup (eager init).
  • Lazy initialization (@Lazy): Bean created only when needed.
  • Example: 
    @Component
@Lazy
public class HeavyBean { ... }
  • Useful for resource-heavy or optional beans.

3. Spring REST Controllers

  • A REST controller handles HTTP requests and returns responses in JSON/XML.
  • @RestController = @Controller + @ResponseBody
  • Example: 
    @RestController
@RequestMapping("/users")
public class UserController {
     @GetMapping
    public List<User> getAllUsers() { ... }
}

  • @RequestMapping: General mapping for all HTTP methods.
  • Shorthand annotations (preferred in REST APIs):
    • @GetMapping → GET (fetch data).
    • @PostMapping → POST (create resource).
    • @PutMapping → PUT (update resource).
    • @DeleteMapping → DELETE (remove resource).
  • Example: 
    @GetMapping("/{id}")
public User getUserById(@PathVariable Long id) { ... }

  • Request Parameters (@RequestParam): Extracts query params from URL.
    Example: /users?active=true 
    @GetMapping
public List<User> getUsers(@RequestParam boolean active) { ... }
  • Path Variables (@PathVariable): Extracts values from URL path.
    Example: /users/5 
    @GetMapping("/{id}")
public User getUser(@PathVariable Long id) { ... }

  • @RequestBody: Maps JSON request body to Java object. 
    @PostMapping
public User createUser(@RequestBody User user) { ... }
  • ResponseEntity: Customize HTTP response (status, headers, body). 
    return ResponseEntity.status(HttpStatus.CREATED).body(user);

  • @ControllerAdvice: Global exception handler across controllers.
  • @ExceptionHandler: Handles specific exceptions.
  • Example: 
    @ControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<String> handleUserNotFound(UserNotFoundException ex) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(ex.getMessage());
    }
}
  • Ensures consistent error responses instead of exposing raw exceptions.

  • DTO (Data Transfer Object): A custom class used to transfer data between client and server.
  • Benefits:
    • Hides internal entity structure.
    • Helps with validation & transformation.
    • Keeps response clean and client-friendly.
  • Example: 
    public class UserDTO {
    private String name;
    private String email;
}
  • Controller returns DTO instead of entity: 
    return new UserDTO(user.getName(), user.getEmail());

4. Spring Data JPA

  • Simplifies data access using JPA (Java Persistence API).
  • Removes boilerplate DAO code by providing repositories.
  • Integrates with Hibernate (default JPA provider).
  • Benefits:
    • Auto-implemented CRUD operations.
    • Powerful query creation from method names.
    • Supports pagination, sorting, transactions.

  • @Entity: Marks class as a JPA entity (maps to table).
  • @Table: Specifies table details (optional if class name matches).
  • Example: 
    @Entity
@Table(name = "users")
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
            
    @Column(nullable = false)
     private String name;
}

  • CrudRepository<T, ID>: Basic CRUD methods.
  • JpaRepository<T, ID>: Extends CrudRepository, adds pagination & batch operations.
  • Example: 
    public interface UserRepository extends JpaRepository<User, Long> {}

  • Create queries by defining method names.
  • Examples: 
    List<User> findByName(String name);
List<User> findByEmailContaining(String keyword);
List<User> findByAgeGreaterThan(int age);
  • Spring parses method names & generates SQL automatically.

  • JPQL (Java Persistence Query Language): Object-oriented queries using entity names. 
    @Query("SELECT u FROM User u WHERE u.name = :name")
List<User> findByName(@Param("name") String name);
  • Native Queries: Raw SQL queries directly. 
    @Query(value = "SELECT * FROM users WHERE email = ?1", nativeQuery = true)
User findByEmailNative(String email);

  • Spring Data JPA provides Pageable and Sort.
  • Example: 
    Page<User> findAll(Pageable pageable);
  • Usage: 
    Pageable pageable = PageRequest.of(0, 10, Sort.by("name").ascending());
Page<User> users = userRepository.findAll(pageable);
  • Benefits:
    • Avoids loading large datasets at once.
    • Useful for APIs returning limited results.

  • @Transactional: Ensures DB operations run as one unit (commit/rollback).
  • Example: 
    @Service
public class UserService {
    @Transactional
    public void registerUser(User user) {
        userRepository.save(user);
        emailService.sendWelcomeEmail(user); 
        // rolled back if save fails
    }
}
  • Benefits:
    • Maintains data consistency.
    • Prevents partial updates on errors.

5. Database Setup & Migrations

  • MySQL & PostgreSQL:
    • Popular relational DBs supported by Spring Data JPA.
    • Add driver dependency (mysql-connector-java or postgresql).
    • Configure application.properties: 
      spring.datasource.url=jdbc:mysql://localhost:3306/mydb
spring.datasource.username=root
spring.datasource.password=secret
spring.jpa.hibernate.ddl-auto=update
    • Spring Boot auto-configures DataSource, EntityManager, and TransactionManager.
  • H2 Database:
    • In-memory DB for testing & prototyping.
    • Dependency: com.h2database:h2.
    • Auto-starts with Spring Boot → Access at /h2-console.
    • Great for unit tests.

  • Add spring-boot-starter-data-mongodb dependency.
  • Configure: 
    spring.data.mongodb.uri=mongodb://localhost:27017/mydb
  • Create entity with @Document instead of @Entity: 
    @Document(collection = "users")
public class User {
    @Id
    private String id;
    private String name;
}
  • Repository: 
    public interface UserRepository extends MongoRepository<User, String> {}
  • Benefits:
    • No schema migrations needed.
    • Great for flexible, JSON-like data structures.

  • Why Migrations? → Manage schema changes consistently, version-controlled.
  • Flyway:
    • Uses SQL scripts → place in src/main/resources/db/migration.
    • Example V1__init.sql: 
      CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(100)
);
    • Applied automatically on startup.
  • Liquibase:
    • Uses XML, YAML, JSON, or SQL changesets.
    • Example (db.changelog-master.xml): 
      <changeSet id="1" author="admin">
    <createTable tableName="users">
        <column name="id" type="BIGINT" primaryKey="true"/>
        <column name="name" type="VARCHAR(100)"/>
    </createTable>
</changeSet>
    • Supports rollbacks, branching, multiple environments.
  • Comparison:
    • Flyway → Simple, SQL-based, quick setup (best for smaller projects).
    • Liquibase → Flexible, enterprise-grade, supports multiple formats.

6. Validation & Data Binding

  • Spring Boot integrates with Jakarta Bean Validation (JSR-380) → Hibernate Validator by default.
  • Common annotations:
    • @NotNull → field must not be null
    • @NotBlank → string must not be null/empty
    • @Size(min=, max=) → limits string/collection size
    • @Email → valid email format
    • @Min, @Max → numeric ranges
  • Example DTO: 
    public class UserDTO {
    @NotBlank
    private String name;
            
    @Email
    private String email;
            
    @Min(18)
    private int age;
}
  • Controller usage: 
    public ResponseEntity<?> createUser(@Valid @RequestBody UserDTO dto) { ... }

  • For domain-specific rules (e.g., phone numbers, usernames).
  • Step 1 → Create annotation: 
    @Constraint(validatedBy = PhoneNumberValidator.class)
public @interface ValidPhoneNumber {
    String message() default "Invalid phone number";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}
  • Step 2 → Implement validator: 
    public class PhoneNumberValidator 
        implements ConstraintValidator<ValidPhoneNumber, String> {
    public boolean isValid(String value, ConstraintValidatorContext context) {
        return value != null && value.matches("\\d{10}");
    }
}

  • Data Binding: Spring auto-maps request JSON/params → DTOs.
  • Type Conversion:
    • Built-in: String → int, String → Date, etc.
    • Custom: Implement Converter<S, T> or use @InitBinder.
  • Example custom converter: 
    public class StringToLocalDateConverter 
        implements Converter<String, LocalDate> {
    public LocalDate convert(String source) {
        return LocalDate.parse(source);
    }
}

  • When validation fails → Spring throws MethodArgumentNotValidException.
  • Best practice → format structured JSON errors with @ControllerAdvice.
  • Example: 
    @ControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, String>> handleValidationErrors(
              MethodArgumentNotValidException ex) {
        Map<String, String> errors = new HashMap<>();
        ex.getBindingResult().getFieldErrors()
          .forEach(error -> errors.put(error.getField(), error.getDefaultMessage()));
        return ResponseEntity.badRequest().body(errors);
    }
}
  • Client receives clean JSON: 
    {
  "name": "must not be blank",
  "email": "must be a well-formed email address"
}

7. Configuration & Profiles

  • Spring Boot allows configuring application settings through application.properties or application.yml in src/main/resources.

  • Common configurations include:

    • Server: server.port=8081
    • Database: spring.datasource.url=...
    • Logging: logging.level.org.springframework=DEBUG

  • application.properties → Key-value pairs.

    server.port=8081
spring.datasource.username=root

    application.yml → Hierarchical, easier for complex configs.

    server:
  port: 8081
spring:
  datasource:
    username: root

  • Both are functionally equivalent; .yml is often preferred for readability.

  • Profiles allow you to define different configurations for different environments (dev, test, prod).

  • Profile-specific files:

    • application-dev.yml
    • application-test.yml
    • application-prod.yml

  • Activate profile:

    # In properties
    spring.profiles.active=dev
                
    # Or via command line
    java -jar app.jar --spring.profiles.active=prod

    @Profile annotation example:

    @Service
    @Profile("dev")
    public class DevEmailService implements EmailService {
        ...
    }

  • Only active in the dev profile.

  • Benefit: Separates configurations (DB credentials, logging, cache) for different environments.

  • Spring Boot allows configs to be externalized so the same code can run in multiple environments without changes.

  • Sources of configuration (in order of precedence):
    1. Command-line arguments.
    2. Environment variables (e.g., SPRING_DATASOURCE_URL).
    3. application.properties / application.yml.
    4. Default values in code.

  • Example using environment variables: 
    spring.datasource.username=${DB_USER:root}
spring.datasource.password=${DB_PASS:password}

  • Reads from environment variables (DB_USER, DB_PASS), defaults to root/password if not set.

  • Benefits:
    • Secure (no hardcoding secrets).
    • Easy to override in CI/CD pipelines, Docker, Kubernetes.

8. Spring Security

  • Spring Security is the security framework for Spring apps.

  • Provides:

    • Authentication → Verifying identity (who you are).
    • Authorization → Checking permissions (what you can do).
    • Protection against attacks like CSRF, XSS, Session fixation.

  • Integration with Spring Boot: spring-boot-starter-security.

  • By default, all endpoints are secured with basic authentication and a generated password.

  • Authentication: Verifies user credentials (username/password, token, etc.). Implemented via AuthenticationManager and UserDetailsService.
  • Authorization: Grants/denies access based on roles/permissions. Example: Only admins can delete a user.
  • Flow:
    1. User logs in → credentials verified.
    2. On success → SecurityContext stores authenticated user.
    3. Authorization checks run on each request.

  • RBAC assigns permissions based on roles (e.g., ADMIN, USER).

  • Example: 
    http
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/admin/**").hasRole("ADMIN")
    .requestMatchers("/user/**").hasAnyRole("USER", "ADMIN")
    .anyRequest().authenticated()
);
  • Benefits:
    • Easier to manage permissions.
    • Widely used in enterprise applications.

  • JWT is a stateless token-based authentication method.

  • Flow:
    1. User logs in with credentials.
    2. Server issues a JWT signed with a secret key.
    3. Client sends JWT in Authorization: Bearer <token> header.
    4. Server validates token (no DB lookup needed).

  • Benefits: Scalable for microservices, stateless.

  • Example claims in JWT payload:

    {
  "sub": "user123",
  "role": "ADMIN",
  "exp": 1699999999
}

  • OAuth2: Authorization framework for delegated access (e.g., login with Google/GitHub).
  • OpenID Connect (OIDC): Adds authentication (who the user is) on top of OAuth2 → supports SSO.

  • Spring Boot provides spring-boot-starter-oauth2-client and spring-security-oauth2-resource-server for easy integration.

  • Enables fine-grained authorization at the method level.
    • @Secured: Allows access only if the user has a specific role. 
      @Secured("ROLE_ADMIN")
    public void deleteUser(Long id) { ... }
    • @PreAuthorize: More powerful, supports SpEL. 
      @PreAuthorize("hasRole('ADMIN') or #id == authentication.principal.id")
    public User getUser(Long id) { ... }

  • Benefits: Security rules are close to business logic and more flexible than URL-based restrictions.


9. Spring Boot Testing

  • Unit testing: Testing individual classes/methods in isolation.
    • JUnit: Standard Java testing framework.
      • @Test → Marks a test method.
      • @BeforeEach / @AfterEach → Setup and cleanup before/after each test.
    • Example: 
      @Test
void testAddUser() {
    UserService service = new UserService();
    User user = service.addUser("John");
    assertEquals("John", user.getName());
}
    • Mockito: Mock dependencies to isolate the unit being tested. 
      @Mock
UserRepository repo;
                
@InjectMocks
UserService service;
                
@Test
void testAddUser() {
    when(repo.save(any(User.class))).thenReturn(new User("John"));
    User user = service.addUser("John");
    assertEquals("John", user.getName());
}

  • Integration testing: Testing multiple components together (controllers, services, repositories).

  • @SpringBootTest: Boots up the full Spring context for testing.
  • Example: 
    @SpringBootTest
class UserServiceIntegrationTest {
    @Autowired
    UserService service;
                
    @Test
    void testAddUserIntegration() {
        User user = service.addUser("Alice");
        assertNotNull(user.getId());
    }
}
  • Can be combined with @Transactional to rollback DB changes after tests.

  • MockMvc: Simulates HTTP requests without starting a real server. Useful for controller layer testing.

    @AutoConfigureMockMvc
@SpringBootTest
class UserControllerTest {
    @Autowired
    private MockMvc mockMvc;
                
    @Test
    void testGetUser() throws Exception {
        mockMvc.perform(get("/users/1"))
                .andExpect(status().isOk())
                .andExpect(jsonPath("$.name").value("John"));
    }
}

  • Testcontainers: Provides temporary Docker containers for databases during tests. Ensures a real DB environment without affecting local setup.

  • Example for PostgreSQL:

    @Testcontainers
@SpringBootTest
class UserRepositoryTest {
                
    @Container
    static PostgreSQLContainer postgres = new PostgreSQLContainer<>("postgres:15")
        .withDatabaseName("testdb")
        .withUsername("user")
        .withPassword("pass");
                
    @Autowired
    UserRepository repo;
                
    @Test
    void testSaveUser() {
        User user = new User("Alice");
        repo.save(user);
        assertEquals(1, repo.findAll().size());
    }
}

  • Benefits: Tests run in isolated, real DB environments; works well with CI/CD pipelines.


10. Spring Boot Actuator & Advanced Features

  • Spring Boot Actuator provides production-ready monitoring for applications.

  • Adds endpoints to check app health, metrics, environment, and more.
  • Example dependency: spring-boot-starter-actuator
  • Common endpoints:
    • /actuator/health → Application health
    • /actuator/metrics → Metrics like memory, CPU, requests
    • /actuator/env → Environment properties

  • Health checks verify that services (DB, cache, messaging) are running.
  • Customize health indicators: 
    @Component
public class CustomHealthIndicator implements HealthIndicator {
    @Override
    public Health health() {
        return Health.up().withDetail("status", "All good").build();
    }
}
  • Metrics: Track performance, request counts, DB connections, etc. (Micrometer integration).

  • @Async allows methods to run asynchronously in a separate thread.
  • Useful for long-running tasks like sending emails.
  • Example: 
    @Async
public void sendEmail(User user) { ... }
  • Enable async support: 
    @EnableAsync
@SpringBootApplication

  • @Scheduled allows running methods at fixed intervals or cron expressions.
  • Examples: 
    @Scheduled(fixedRate = 5000)
public void checkDatabase() { ... }
            
@Scheduled(cron = "0 0 * * * ?")
public void runDailyTask() { ... }
  • Enable scheduling: 
    @EnableScheduling
@SpringBootApplication

  • Declarative caching using @Cacheable and @CacheEvict.
  • Example: 
    @Cacheable("users")
public User getUser(Long id) { ... }
            
@CacheEvict(value = "users", key = "#id")
public void updateUser(Long id, User user) { ... }
  • Supports multiple providers: EhCache, Redis, Caffeine.
  • Benefits: Reduces DB load, improves performance.

  • Multipart file upload using MultipartFile.
  • Example upload: 
    @PostMapping("/upload")
public String uploadFile(@RequestParam MultipartFile file) { ... }
  • Example download: 
    @GetMapping("/download/{filename}")
public ResponseEntity<Resource> downloadFile(@PathVariable String filename) { ... }
  • Useful for storing files in filesystem, DB, or cloud storage.

  • Supports multiple languages using messages.properties and messages_{locale}.properties.
  • Configure LocaleResolver: 
    @Bean
public LocaleResolver localeResolver() {
    SessionLocaleResolver slr = new SessionLocaleResolver();
    slr.setDefaultLocale(Locale.ENGLISH);
    return slr;
}
  • Access messages via @Autowired MessageSource messageSource.

  • Enables real-time communication between client and server.
  • Use spring-boot-starter-websocket.
  • Example: 
    @Controller
public class ChatController {
    @MessageMapping("/chat")
    @SendTo("/topic/messages")
    public Message sendMessage(Message msg) { return msg; }
}
  • Supports STOMP protocol, enabling chat apps, live notifications, or dashboards.

11. Microservices Architecture & Spring Cloud

  • Microservices: Application broken into small, independent services communicating over APIs.
  • Characteristics:
    • Each service is independently deployable
    • Own database or data storage
    • Focused on a single business capability
  • Benefits: Scalability, flexibility, easier maintenance, faster deployments.
  • Contrast with Monolithic:
    • Monolithic → Single deployable, tightly coupled
    • Microservices → Loosely coupled, independently deployable

  • Spring Cloud provides tools for building microservices with Spring Boot.
  • Eureka (Service Discovery):
    • Registers microservices for dynamic discovery
    • Eliminates hard-coded service URLs
  • Config Server: Centralizes configuration for all services (Git, filesystem, etc.)
  • API Gateway (Spring Cloud Gateway / Zuul):
    • Single entry point for all clients
    • Handles routing, rate limiting, authentication, logging

  • Services communicate via HTTP APIs or messaging queues.
  • Clients:
    • RestTemplate: Synchronous HTTP client
    • WebClient: Reactive, non-blocking HTTP client (Spring WebFlux)
    • Feign Client: Declarative REST client integrated with Eureka
  • Example Feign client: 
    @FeignClient(name = "user-service")
public interface UserClient {
    @GetMapping("/users/{id}")
    User getUser(@PathVariable Long id);
}
  • Choice depends on sync vs async needs and reactive programming.

  • Provides circuit breakers, retries, rate limiting, bulkhead patterns.
  • Prevents cascading failures in microservices.
  • Example: 
    @CircuitBreaker(name = "userService", fallbackMethod = "fallbackUser")
public User getUser(Long id) { ... }
            
public User fallbackUser(Long id, Throwable ex) {
    return new User("default");
}
  • Improves system reliability when dependent services fail.

  • Tracks flow of a request across multiple services.
  • Tools:
    • Spring Cloud Sleuth: Adds trace IDs to logs
    • Zipkin: Collects traces and visualizes request flow
  • Helps debug latency issues and identify bottlenecks.
  • Example log with Sleuth: 
    [traceId=abc123 spanId=xyz456] GET /orders/1

12. Spring Boot Deployment & Production Readiness

  • Executable Jar (default) → Contains embedded server (Tomcat/Jetty) 
    mvn clean package
java -jar target/app.jar
  • War → Deployable to external servlet containers
  • Benefits of executable jar:
    • No need for separate app server
    • Simplifies deployment and CI/CD pipelines

  • Containerizing apps ensures environment consistency across dev, test, prod.
  • Dockerfile example: 
    FROM openjdk:17-jdk-slim
COPY target/app.jar app.jar
ENTRYPOINT ["java","-jar","/app.jar"]
  • Build & run: 
    docker build -t myapp:1.0 .
docker run -p 8080:8080 myapp:1.0
  • Benefits: Easy scaling, consistent runtime, easier orchestration (Kubernetes)

  • Cloud Platforms:
    • AWS: Elastic Beanstalk, ECS, Lambda (serverless)
    • Azure: App Service, Kubernetes Service
    • GCP: App Engine, Cloud Run, Kubernetes Engine
  • Typical deployment steps:
    • Build jar/war or Docker image
    • Upload to cloud environment
    • Configure environment variables, database connections, scaling

  • SLF4J: Logging facade; allows switching between implementations (Logback, Log4j2)
  • Logback: Default logging implementation in Spring Boot
  • Configuration example: 
    logging.level.org.springframework=INFO
logging.file.name=app.log
  • Benefits:
    • Structured logging
    • Separate logs for production, development
    • Helps with monitoring, debugging, auditing

  • Key areas:
    • Connection Pooling: Use HikariCP
    • Caching: Spring Cache or Redis
    • Async Processing: Offload tasks with @Async
    • Proper indexing & query optimization for DB
    • Monitoring & Metrics: Actuator & Micrometer
    • Thread pool tuning for web requests
  • Benefits: Ensures scalable, fast, and reliable production apps
