1. Fundamentals of Testing

Progress:

  • Correctness: Ensures the software meets requirements. Example: Checking if login works with valid credentials.
  • Reliability: Ensures the software works consistently under expected conditions. Example: App doesn’t crash when thousands of users log in daily.
  • Performance: Ensures the software runs efficiently (speed, scalability, resource usage). Example: Website loads within 2 seconds even under heavy traffic.

  • Manual vs Automated
    • Manual testing: Executed by humans. Good for exploratory/usability/small projects. Slower, prone to errors.
    • Automated testing: Uses scripts/tools. Good for regression, performance, large-scale testing. Faster, repeatable, requires setup/maintenance.
  • Functional vs Non-functional
    • Functional: What the system does. Verifies features against requirements. Examples: unit tests, integration tests, UAT.
    • Non-functional: How the system works. Tests performance, security, usability, scalability. Example: load testing for response time under stress.
  • White-box vs Black-box vs Grey-box
    • White-box: Tester knows internal code/logic. Example: unit tests by developers.
    • Black-box: Tester only knows inputs/outputs, not code. Example: QA testing login with different usernames/passwords.
    • Grey-box: Tester has partial knowledge. Example: Security testing with DB knowledge but not full code.

2. Unit Testing

Progress:

  • Unit testing focuses on testing the smallest parts of the code (functions, methods, or classes) in isolation.
  • Example: Testing a function that calculates the sum of two numbers without involving the database or UI.
  • Ensure each small unit works correctly before combining them into a bigger system.

  • Best practices:
    • Keep functions small and focused on one task.
    • Use clear inputs and outputs (avoid hidden dependencies).
    • Avoid hardcoding values (use parameters/configs).
    • Use Dependency Injection to make replacing real services with mocks easier.

  • Java:
    • JUnit → Most common testing framework, provides annotations like @Test for easy test methods.
    • TestNG → More advanced, supports parallel tests and better test configuration.
  • JavaScript:
    • Jest → Widely used with React, supports mocking and snapshot testing.
    • Mocha → Flexible test framework, often used with Chai (assertion library).
    • Jasmine → BDD-style testing, older but still used in Angular.
  • Python:
    • unittest → Built-in framework.
    • pytest → Popular because of simple syntax and powerful plugins.

  • Sometimes, a unit depends on external systems (database, API, files).
  • Mocking replaces these real dependencies with fake ones, so tests stay fast and independent.
  • Examples:
    • Java: Mockito (most common mocking framework).
    • JavaScript: Sinon (for mocking, stubbing, and spying).
    • Python: unittest.mock (standard library for mocking).

3. Integration Testing

Progress:

  • Integration testing checks whether different parts of the application (modules, services, or components) work correctly when combined.
  • Example: If you have a user service and an authentication service, integration testing ensures that a user created in the user service can successfully log in via the authentication service.

  • Many failures in real systems occur at the boundaries where modules interact.
  • Integration testing focuses on catching issues such as:
    • Incorrect data formats between APIs.
    • Wrong method calls or parameter mismatches.
    • Network/communication errors in distributed systems.
  • Example: If a frontend expects userEmail but the backend API returns emailId, integration tests will detect this mismatch.

  • Spring Boot Test (Java):
    • Provides annotations like @SpringBootTest to load the application context and test how beans, services, and repositories interact.
    • Can test REST endpoints and database interactions in a real or in-memory environment.
  • Supertest (Node.js):
    • A library for testing HTTP APIs built with frameworks like Express.js.
    • Allows you to send requests (GET, POST, PUT, DELETE) and assert responses.
  • Postman / Newman (API integration):
    • Postman is a GUI tool for manually testing APIs.
    • Newman is the CLI companion that can run Postman test collections automatically in CI/CD pipelines.
    • Useful for end-to-end API flow testing across services.

4. End-to-End (E2E) Testing

Progress:

  • E2E testing verifies the entire user journey in an application, from start to finish.
  • Example: For an e-commerce app, E2E testing would cover logging in, searching for a product, adding it to the cart, and completing checkout.
  • Ensure the system works as expected in a real-world scenario.

  • Unlike unit or integration testing (which focus on smaller parts), E2E tests check how frontend, backend, databases, APIs, and third-party services work together.
  • Helps catch issues like:
    • Misconfigured APIs
    • Broken UI flows
    • Incorrect data handling

  • Selenium:
    • Widely used for automating browser actions.
    • Supports multiple languages (Java, Python, JavaScript).
    • Great for cross-browser testing.
  • Cypress:
    • Modern JavaScript-based testing tool.
    • Fast execution with real-time reloading.
    • Popular for testing React, Angular, and Vue apps.
  • Playwright:
    • Developed by Microsoft.
    • Supports multiple browsers (Chromium, Firefox, WebKit).
    • Can handle modern features like multi-tab testing and network mocking.

5. Regression Testing

Progress:

  • Regression testing verifies that new code changes (like adding features or fixing bugs) do not unintentionally cause issues in parts of the system that were previously working.
  • Example: You fix a login bug, but regression testing ensures the signup, password reset, and dashboard still work correctly.

  • Since regression testing needs to be repeated frequently, automation is highly effective.
  • Automated regression suites are a collection of tests that run automatically whenever changes are made.
  • Tools: JUnit/TestNG (Java), Jest (JavaScript), Pytest (Python), Selenium/Cypress (UI).

  • Regression testing is often integrated into CI/CD pipelines so tests run automatically whenever code is pushed, built, or deployed.
  • This ensures that issues are caught early, before reaching production.
  • Example: In a GitHub Actions pipeline, every pull request triggers the regression test suite to validate changes.

6. Performance & Load Testing

Progress:

  • Performance testing checks how fast, stable, and efficient the system is under different workloads.
  • Load testing checks how the system behaves when many users access it at the same time.
  • Scalability testing ensures the system can handle increased load by adding resources (e.g., more servers).
  • Example: Making sure an e-commerce website works smoothly on Black Friday when traffic spikes.

  • JMeter: Widely used open-source tool for simulating heavy loads on web servers and analyzing performance.
  • Gatling: Focuses on load and stress testing, written in Scala, great for automating performance tests in CI/CD.
  • Locust: Python-based tool that lets you write load tests in code, making it flexible and easy to scale with distributed users.

  • Response time: How long it takes for the system to respond to a request.
  • Throughput: Number of requests the system can handle per second/minute.
  • Latency: Delay between sending a request and receiving the first response.
  • Resource usage: CPU, memory, disk, and network utilization under load.
  • Note: These metrics help identify bottlenecks, like slow queries or insufficient server capacity.

7. Security & Vulnerability Testing

Progress:

  • Security testing ensures the system is protected from potential attacks and data breaches.
  • Code-level checks often include input validation, avoiding hardcoded credentials, proper authentication, and safe handling of sensitive data.
  • Example: Preventing SQL Injection by using parameterized queries instead of string concatenation.

  • Penetration testing (pen testing) simulates real-world attacks to find weaknesses before hackers do.
  • It can involve testing login systems, APIs, session handling, and access controls to ensure no unauthorized access is possible.
  • Example: Trying to access admin pages without proper login to test if the system blocks it.

  • OWASP ZAP: Free, open-source tool for finding vulnerabilities in web applications (SQL injection, XSS, etc.).
  • Burp Suite: Popular commercial tool for manual and automated penetration testing, offering features like proxy interception, scanning, and exploitation.

8. Test Automation & CI/CD

Progress:

  • Automated test suites are collections of tests that run without manual intervention.
  • They can cover unit, integration, regression, and E2E tests to ensure consistent quality.
  • Example: Writing a Jest suite for a React app that tests login, signup, and logout flows automatically.

  • Automated tests are integrated into CI/CD pipelines to run on every code change.
  • This ensures that new commits or deployments do not break existing functionality.
  • Example: Using GitHub Actions to run unit and integration tests every time a pull request is opened.

  • Run tests frequently on code changes.
  • Prioritize fast-running tests in CI to get quick feedback.
  • Maintain reliable and stable test suites to avoid false positives/negatives.
  • Monitor and report test results clearly for the team to take action.

9. Best Practices in Testing

Progress:

  • Tests should be easy to understand and update as the code evolves.
  • Best practices:
    • Use clear, descriptive test names (e.g., test_user_can_login_with_valid_credentials).
    • Keep tests small and focused on a single behavior.
    • Follow consistent formatting and naming conventions.
  • Example: Instead of one big test for all user actions, create separate tests for login, signup, and logout.

  • Assertions are used to verify expected outcomes in tests.
  • Effective assertions make tests precise and informative.
  • Best practices:
    • Check specific results, not too broad.
    • Include messages to clarify failures.
  • Example: assertEquals(actualEmail, expectedEmail, "Email should match user input")

  • Test coverage measures how much of your code is exercised by tests (lines, branches, functions).
  • Quality focuses on whether tests catch real issues and verify correct behavior.
  • Best practice: High coverage does not guarantee quality. Aim for meaningful tests that cover critical paths.
  • Example: A test may cover 90% of code but miss edge cases like empty inputs or invalid data.

  • Tests should not depend on or affect each other; each test should run independently.
  • Use mocks, stubs, and setup/teardown methods to isolate environment and dependencies.
  • Example: When testing a payment function, mock the database and API calls so one test doesn’t change the state for others.
