1. Web Security Fundamentals

Progress:

  • A standard awareness document listing the 10 most critical security risks for web applications, published by OWASP.
  • Helps developers prioritize fixing common and dangerous vulnerabilities like injection, XSS, broken authentication.

  • An attack where malicious SQL statements are inserted into queries to manipulate the database.
  • Example: SELECT * FROM users WHERE username = '' OR '1'='1';
  • Prevention: Use prepared statements / parameterized queries, ORM frameworks, and input validation.

  • An attack where a hacker injects malicious JavaScript into a web page that runs in other users’ browsers.
  • Impact: Stealing cookies, hijacking accounts, showing fake content.
  • Prevention:
    • Escape/encode user input before output
    • Use Content Security Policy (CSP)
    • Validate & sanitize inputs

  • Tricks a logged-in user’s browser into sending unintended requests (e.g., transferring money).
  • Example: Malicious site triggers a hidden request to your bank while logged in.
  • Prevention:
    • Use CSRF tokens
    • Enable SameSite cookies
    • Re-authenticate before sensitive actions

  • When apps deserialize untrusted data, attackers can modify it to execute code or escalate privileges.
  • Example: Changing "role":"user" to "role":"admin" in serialized data.
  • Prevention:
    • Avoid deserializing untrusted data
    • Use safe formats (like JSON)
    • Implement integrity checks

  • Weaknesses in login, sessions, and tokens allow attackers to hijack accounts.
  • Examples: Weak passwords, session IDs in URL, no session timeout.
  • Prevention:
    • Use bcrypt or Argon2 for password storage
    • Always use HTTPS
    • Implement session expiration & regeneration

  • HTTP: Data in plain text, vulnerable to eavesdropping/tampering.
  • HTTPS: Uses SSL/TLS for encryption, ensuring confidentiality & integrity.
  • HTTPS prevents MITM attacks and is required for all modern apps.

  • Browser security feature that restricts cross-domain requests.
  • Example: siteA.com calling an API on siteB.com without proper headers → blocked.
  • Key headers:
    • Access-Control-Allow-Origin
    • Access-Control-Allow-Methods
  • Prevention: Set strict CORS policies, never use * in production for sensitive APIs.

2. Authentication & Authorization

Progress:

  • Verifying a user’s identity (who they are).
  • Common methods:
    • Password-based → Username + password.
    • Multi-factor authentication (MFA) → Password + OTP/email/biometric.
    • Token-based → Tokens like JWT or OAuth2 access tokens.
    • Biometric → Fingerprint, face, voice recognition.

  • Password-based: Simple but vulnerable. Needs hashing (bcrypt/Argon2) & MFA.
  • OAuth2: Delegated access protocol (e.g., “Login with Google”). Issues tokens without sharing password.
  • JWT (JSON Web Token): Compact format for stateless authentication.
    Example payload: {"userId":123,"role":"admin"}
  • SSO (Single Sign-On): One login works across multiple apps (e.g., Google for Gmail, YouTube, Drive).

  • Determines what an authenticated user can do.
  • Examples:
    • View vs edit content
    • Admin vs normal user privileges

  • Users are assigned roles; roles define permissions.
  • Example:
    • Admin → manage users
    • Editor → create/edit posts
    • User → read posts
  • Pros: Easy to manage at scale.

  • Access based on user, resource, and environment attributes.
  • Examples:
    • department=HR → access salary data
    • time-based → allow only during work hours
  • Pros: More flexible than RBAC.

  • Sessions store login state; weak management = hijacking risk.
  • Best practices:
    • Use secure, random session IDs
    • Regenerate session IDs on login
    • Store sessions server-side (or secure tokens)

  • Session expiration: End after inactivity or max duration.
  • Secure cookies: Only sent over HTTPS.
  • HttpOnly: Prevents JS from reading cookies (protects against XSS).
  • SameSite:
    • Strict: Only same-site requests
    • Lax: Allows safe cross-site (like GET links)
    • None: Cross-site allowed, but must use Secure

3. Data Security

Progress:

  • Transforming data into a format that unauthorized users cannot read.
  • Protect data at rest (databases, files) and in transit (network).
  • Symmetric and Asymmetric (explained below).

  • Symmetric Encryption:
    • Same key for encryption & decryption
    • Fast and efficient for large data
    • Example: AES (Advanced Encryption Standard)
  • Asymmetric Encryption:
    • Uses key pair: public key (encrypt) & private key (decrypt)
    • Slower but useful for secure communication & digital signatures
    • Example: RSA, ECC

  • Hashing: One-way function converting passwords into fixed-length strings. Cannot be reversed.
  • Salting: Adding random data before hashing to prevent rainbow table attacks.
  • Examples:
    • bcrypt – adaptive, widely used
    • Argon2 – memory-hard, GPU-resistant
  • Best practice: Never store plain passwords; always salt + hash.

  • TLS/SSL: Protocols that encrypt data in transit between client & server.
  • Prevent eavesdropping and tampering.
  • Key points:
    • HTTPS = HTTP + TLS
    • Certificates validate server identity
    • TLS is more secure than older SSL versions

  • Protecting personal & sensitive data from unauthorized access.
  • Principles:
    • Collect only necessary data
    • Limit access to authorized users
    • Protect data at rest & in transit
  • Relevance: Ensures users’ information is confidential.

  • Safely storing, transmitting, and processing sensitive info (passwords, financial data, health records).
  • Best practices:
    • Encrypt sensitive data (AES, RSA)
    • Mask data in logs (avoid full credit card numbers)
    • Use access controls for read/write operations

  • GDPR: EU regulation for user data privacy. Rights: access, erase, correct personal data. Requires consent for processing.
  • HIPAA: US regulation protecting health info. Requires secure storage, access control, audit trails for patient data.

4. Secure Coding Practices

Progress:

  • Input Validation: Check user input is correct, expected, and safe before processing.
  • Example: Ensure email field has valid email format.
  • Prevents: SQL Injection, XSS, buffer overflows.
  • Output Encoding: Convert data before displaying in HTML, JS, or SQL contexts to prevent code execution.
  • Example: Encode <script> tags to &lt;script&gt;.

  • Problem: Functions like eval() in JS, exec() in Python, or dynamic class loading in Java can execute arbitrary code.
  • Risk: Attackers can run malicious commands or scripts.
  • Best Practice: Avoid dynamic code execution; use safe APIs or interpreters that don’t execute raw code.

  • Attackers manipulate file paths to access restricted files.
  • Example: Using ../../ in a URL to access parent directories.
  • Prevention:
    • Validate & normalize file paths
    • Restrict access to specific directories
    • Use APIs that prevent path traversal

  • Risks: Attackers may upload malicious files (scripts, malware).
  • Best Practices:
    • Restrict file types (.jpg, .png)
    • Rename files before saving
    • Scan files for malware
    • Store uploads outside web root; serve via safe URLs

5. API Security

Progress:

  • Ensures only authorized clients can access APIs.
  • JWT (JSON Web Token):
    • Stateless token containing user info (claims) + signature
    • Used for authentication & session management
    • Example: { "userId": 123, "role": "admin" }
  • OAuth2:
    • Delegated access protocol
    • Users allow apps to access resources without sharing passwords
    • Example: “Login with Google” for app API access

  • Rate Limiting: Restrict number of requests per client in a time period
  • Throttling: Slow down or block excessive requests to prevent abuse
  • Protect APIs from DoS attacks & abuse
  • Example: Max 100 requests/minute per user

  • Input Validation: Ensure all API inputs are correct & safe
  • Example: Accept only integers where expected, reject invalid strings
  • Output Sanitization: Encode data before returning to clients to prevent XSS, injection, or script execution

  • Control access to APIs using unique keys per client/app
  • Best Practices:
    • Generate strong, unique keys
    • Rotate & revoke keys as needed
    • Store keys securely (never hardcode in client apps)
    • Monitor usage & track abuse

6. Network & System Security Basics

Progress:

  • Firewalls: Barrier between trusted & untrusted networks; filter traffic based on IP, port, protocol. Types: Network firewall, WAF.
  • VPN (Virtual Private Network): Encrypts traffic, hides IP, ensures secure remote access.
  • Secure Network Protocols: HTTPS, SSH, SFTP, TLS – encrypt data in transit to prevent eavesdropping & tampering.

  • Proper server setup to reduce vulnerabilities.
  • Best Practices:
    • Disable unused ports & services
    • Apply latest security patches
    • Use strong passwords / SSH keys
    • Limit user privileges (principle of least privilege)
    • Enable logging & monitoring

  • Attacker secretly intercepts or alters communication between two parties.
  • Example: Stealing login credentials over unencrypted Wi-Fi.
  • Prevention:
    • Use HTTPS / TLS for all communication
    • Avoid public Wi-Fi for sensitive transactions
    • Use VPNs for secure remote connections
    • Enable certificate validation to prevent fake servers

7. Monitoring & Logging

Progress:

  • Recording important security-related actions like logins, failed attempts, and admin changes.
  • Best Practices:
    • Ensure logs are tamper-proof and timestamped
    • Store logs securely in a centralized location
    • Monitor logs regularly for unusual patterns

  • Identify potential attacks or anomalies from logs and system behavior.
  • Examples:
    • Multiple failed login attempts (possible brute-force)
    • Unexpected access from unusual IP addresses
    • Sudden data downloads or privilege escalations
  • Tools: SIEM systems, IDS/IPS, automated alerts

  • Logging passwords, credit cards, or tokens can lead to data breaches.
  • Best Practices:
    • Mask sensitive data (****)
    • Avoid logging full tokens or credentials
    • Only log necessary details for troubleshooting or security auditing

8. Security Testing

Progress:

  • Analyzing source code or binaries without executing the program to find security flaws.
  • Detect vulnerabilities like SQL Injection, XSS, insecure configurations early in development.
  • Example Tools: SonarQube, Checkmarx

  • Testing a running application from the outside to find vulnerabilities.
  • Detect runtime issues such as authentication problems, input validation issues, and server misconfigurations.
  • Example Tools: OWASP ZAP, Burp Suite

  • Ethical hacking to simulate real attacks and identify vulnerabilities before attackers do.
  • Evaluate security of networks, apps, and systems.
  • Example Steps: Reconnaissance → Exploitation → Reporting

  • Automated scanning of systems, apps, and networks to identify known vulnerabilities.
  • Quick detection of weak spots like unpatched software, open ports, and misconfigurations.
  • Example Tools: Nessus, Qualys, OpenVAS

9. DevSecOps Practices

Progress:

  • Integrating security checks into the Continuous Integration/Continuous Deployment pipeline.
  • Catch vulnerabilities early before code reaches production.
  • Best Practices:
    • Run automated security tests (SAST, DAST) in the pipeline
    • Enforce code reviews and approval workflows
    • Ensure build servers and deployment environments are secure

  • Monitoring and managing third-party libraries or packages used in your application.
  • Risks: Outdated or vulnerable dependencies can be exploited.
  • Best Practices:
    • Regularly update dependencies
    • Use tools to scan for vulnerabilities (e.g., Dependabot, Snyk, npm audit)
    • Avoid untrusted packages

  • Safely storing sensitive information like API keys, tokens, passwords, and certificates.
  • Best Practices:
    • Use environment variables or secret management tools (e.g., HashiCorp Vault, AWS Secrets Manager)
    • Never commit secrets to version control
    • Rotate secrets regularly
    • Limit access to secrets using role-based access

10. Emerging Topics in Security

Progress:

  • Ensuring security when using cloud services and infrastructure.
  • Best Practices:
    • Use identity and access management (IAM) with least privilege
    • Encrypt data at rest and in transit
    • Enable logging and monitoring (CloudTrail, CloudWatch, etc.)
    • Regularly patch virtual machines and services
    • Use multi-factor authentication (MFA) for accounts

  • Protecting containerized applications and orchestration platforms.
  • Best Practices:
    • Use minimal base images to reduce attack surface
    • Regularly scan images for vulnerabilities (e.g., Trivy, Clair)
    • Limit container privileges (avoid running as root)
    • Use network policies in Kubernetes to control traffic
    • Keep orchestrators and nodes patched

  • Ensuring distributed services communicate securely.
  • Best Practices:
    • Use mutual TLS (mTLS) for service-to-service communication
    • Implement API authentication and authorization
    • Monitor logs centrally to detect suspicious activity
    • Handle secrets securely for each microservice
    • Rate-limit APIs and validate all inputs
