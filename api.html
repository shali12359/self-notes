1. REST Basics

  • REST is an architectural style for designing networked applications, especially web services. It defines a set of principles for creating scalable, simple, and flexible APIs. RESTful APIs use standard HTTP methods and work with resources identified by URLs (Uniform Resource Locators). Each resource can have multiple representations (like JSON, XML, HTML).

  • Client-Server Separation: The client (frontend) and server (backend) are separate entities. The client is responsible for the user interface and user experience, while the server handles data storage and business logic. This separation allows each to evolve independently.
  • Statelessness: Every HTTP request must contain all information needed to understand and process it. The server does not store client context between requests, improving scalability and simplicity.
  • Cacheable Responses: Server responses specify whether they are cacheable, allowing clients or intermediaries to reuse responses and improve performance.
  • Layered System: The architecture can include multiple intermediaries (proxies, gateways) between client and server, without client awareness. This improves scalability and security.
  • Uniform Interface: Interaction between client and server is standardized and consistent, involving resource identification, manipulation through representations, self-descriptive messages, and optionally hypermedia (HATEOAS).
  • Optional: Code on Demand: Servers can optionally send executable code (like JavaScript) to clients to extend functionality temporarily.

  • GET: Retrieve data from the server. Should not modify data. Idempotent and safe.
  • POST: Create a new resource or submit data to the server. Not idempotent.
  • PUT: Update or replace an existing resource completely. Idempotent.
  • PATCH: Partially update an existing resource. Not necessarily idempotent.
  • DELETE: Remove a resource. Idempotent.
  • OPTIONS: Request information about communication options available for a resource (e.g., supported methods). Useful for CORS preflight.
  • HEAD: Similar to GET but only retrieves headers, no body. Useful to check if resource exists or for metadata without full content transfer.

2. API Endpoints & Naming Conventions

  • Use Nouns, Not Verbs
    Endpoints should represent resources (things or entities), not actions. Use nouns to describe what the API exposes.
    Example: Use /users instead of /getUsers or /createUser.
  • Use Plural Names
    Use plural nouns to represent collections of resources. This keeps naming consistent and clear.
    Example: /products for a list of products, /orders for multiple orders.
  • Use Nested Resources for Relationships
    When resources are logically related, nest the dependent resource under the parent resource to show hierarchy.
    Example: /users/{id}/orders to represent orders belonging to a specific user.
  • Use Query Parameters for Filtering and Sorting
    Use query parameters to filter, sort, or paginate collections without adding complexity to the path.
    Example: /products?category=books&sort=price filters products by category and sorts by price.
  • Use Path Parameters to Identify Specific Resources
    Use path parameters to specify individual resources clearly.
    Example: /users/{id} where {id} is a placeholder for the user ID.
  • Avoid Deep Nesting
    Deeply nested URLs can become hard to read and maintain. Limit nesting to 2 levels generally. If more complex relationships exist, consider other approaches like query parameters or separate endpoints.
    Example: Avoid /users/{userId}/orders/{orderId}/items/{itemId} if possible.

  • URI Versioning
    Include the API version in the URI path to clearly separate versions. Easy to understand and use.
    Example: /v1/users, /v2/users
  • Header-Based Versioning
    Specify the version in a custom request header (e.g., Accept header). Keeps URLs clean but requires clients to manage headers correctly.
    Example: Accept: application/vnd.example.v1+json
  • Query Parameter Versioning
    Pass the version as a query parameter in the URL. Simple but less explicit than URI versioning.
    Example: /users?version=1

3. HTTP Status Codes

  • Definition: The client’s request was successfully received, understood, and accepted.
  • Common Examples:
    • 200 OK: Request succeeded and response contains requested data.
    • 201 Created: Resource successfully created (usually after POST).
    • 204 No Content: Request succeeded but no content to return (e.g., after DELETE).

  • Definition: Further action needed by the client to complete the request, usually to follow a redirect.
  • Common Examples:
    • 301 Moved Permanently: Resource permanently moved to new URL.
    • 302 Found: Resource temporarily resides under a different URL.
    • 304 Not Modified: Resource has not changed; client can use cached version.

  • Definition: Errors caused by invalid or incorrect client requests.
  • Common Examples:
    • 400 Bad Request: Invalid syntax in request.
    • 401 Unauthorized: Authentication required or failed.
    • 403 Forbidden: Authenticated but no permission.
    • 404 Not Found: Resource does not exist.
    • 409 Conflict: Request conflicts with current resource state.

  • Definition: Server failed to fulfill a valid request due to internal issues.
  • Common Examples:
    • 500 Internal Server Error: Generic unexpected server error.
    • 502 Bad Gateway: Invalid response from upstream server.
    • 503 Service Unavailable: Server overloaded or down for maintenance.
    • 504 Gateway Timeout: No timely response from upstream server.

4. Request & Response Formats

  • Definition: JSON (JavaScript Object Notation) is the most common format for sending and receiving data in REST APIs.
  • Why it’s used: Lightweight, human-readable, easy for machines to parse, supported by most programming languages.
  • Example: 
    {
    "id": 1,
    "name": "John Doe",
    "email": "john@example.com"
}

  • Definition: XML (Extensible Markup Language) is another format for data exchange.
  • Why it’s used: Useful for integrating with legacy systems or systems requiring structured markup.
  • Example: 
    <user>
    <id>1</id>
    <name>John Doe</name>
    <email>john@example.com</email>
</user>
  • Note: JSON is preferred today, but XML support can be optional.

  • Content-Type: Tells the server the format of the request body.
    Example: Content-Type: application/json, Content-Type: application/xml
  • Accept: Tells the server the format expected in the response.
    Example: Accept: application/json, Accept: application/xml
  • Authorization: Sends authentication credentials so the API can verify the client.
    Example: Authorization: Bearer <token>, Authorization: Basic <base64encoded>

  • Pagination Formats: Manage large datasets by splitting them into smaller chunks.
  • Limit & Offset:
    Definition: limit = number of items to return, offset = how many items to skip.
    Example: /products?limit=10&offset=20 → 10 products starting from the 21st.
  • Page & Size:
    Definition: page = which page of results, size = number of items per page.
    Example: /products?page=3&size=10 → 10 products from page 3.
  • Best Practice: Always return pagination metadata (total items, total pages, current page).

5. Error Handling

  • Definition: All API error responses should follow a standard structure so clients can easily parse and handle them.
  • Example: 
    {
    "timestamp": "2025-08-09T12:34:56Z",
    "status": 400,
    "error": "Bad Request",
    "message": "Invalid email format",
    "path": "/users"
}
  • Why: Improves debugging, ensures predictable client-side behavior, and makes integration smoother.

  • Definition: Create specific exception classes to represent different error scenarios in your application.
  • Why: Helps differentiate between types of errors (e.g., UserNotFoundException, InvalidInputException) and makes code more readable and maintainable.

  • Definition: Returned when the client sends data that fails validation rules.
  • Example: Missing required fields, invalid formats, or out-of-range values.
  • Why: Clearly informs the client what’s wrong so they can fix the request.

  • Spring Boot: @ControllerAdvice
    Definition: A class annotated with @ControllerAdvice can catch and handle exceptions globally across all controllers.
    Why: Centralizes error handling, avoids duplicate try/catch blocks.
  • NestJS: ExceptionFilter
    Definition: A class implementing ExceptionFilter can intercept and format exceptions across the application.
    Why: Ensures consistent error responses and clean controller code.

6. Authentication & Authorization

Authentication method where the server issues a token after verifying user credentials.
  • Token is sent in subsequent requests, usually in the Authorization header.
  • Stateless – server does not store session data.
  • 1. JWT (JSON Web Token)
    • A compact, URL-safe token format containing claims about the user.
    • Structure: Header, Payload, Signature.
    • Commonly used for API authentication.
    • Signed to ensure integrity, can be verified without contacting the issuer.
  • 2. Refresh tokens
    • Used to obtain a new access token without re-authenticating the user.
    • Stored securely (not in local storage).
    • Helps maintain long-lived sessions.
    • Usually has longer expiration than access tokens.

Open standard for delegated authorization.
  • Allows applications to access resources on behalf of a user.
  • Common flows: Authorization Code, Client Credentials, Implicit, Password.
  • Used by platforms like Google, Facebook for third-party login.

Simple token used to identify and authenticate an application or user.
  • Often sent in request headers or query parameters.
  • Should be kept secret and rotated periodically.
  • Not as secure as JWT or OAuth2 if exposed.

Access control method based on user roles.
  • Roles define permissions to perform certain actions.
  • Common in enterprise applications.
  • Example: Admin can delete users, while User cannot.

Mechanism to allow restricted resources on a web page to be accessed from another domain.
  • Access-Control-Allow-Origin specifies allowed origins.
  • Prevents unauthorized cross-domain requests.
  • Important for securing APIs consumed by browsers.

7. API Pagination, Sorting, Filtering & HATEOAS

  • What it is: Pagination means splitting large sets of results into smaller, more manageable pages.
  • How it works: Send query parameters in the request URL to control which page and how many items per page are returned.
  • Common parameters:
    • page → Which page to fetch (starting from 1 or 0 depending on design).
    • size → Number of items per page.
  • Example: GET /products?page=2&size=20 → Fetch the second page of products, each page having 20 items.
  • Benefits:
    • Reduces server load and response size.
    • Improves client performance.
    • Helps in infinite scrolling or "load more" patterns.

  • What it is: Sorting controls the order of returned results.
  • How it works: Pass a sort parameter in field,direction format.
  • Direction options:
    • asc → Ascending order
    • desc → Descending order
  • Examples:
    • GET /products?sort=price,asc → Products sorted by price ascending.
    • GET /users?sort=lastName,desc → Users sorted by last name descending.
    • GET /products?sort=category,asc&sort=price,desc → Multiple sorting fields.

  • What it is: Filtering lets clients narrow results to only those matching specific criteria.
  • How it works: Query parameters represent filters, with each filter key matching a field in the dataset.
  • Examples:
    • GET /orders?status=active → Only active orders.
    • GET /products?category=books&priceLessThan=20 → Books under $20.
    • GET /users?ageGt=18&country=US → Advanced filtering with operators like gt, lt, in.

  • What it is: A REST principle where the API provides links in responses to guide the client on possible next actions.
  • Why: Makes APIs more discoverable and removes the need for clients to hardcode URL paths.
  • How it works: The API returns hypermedia links (often in _links field) along with resource data.
  • Example response: 
    {
  "id": 123,
  "title": "REST API Design",
  "status": "active",
  "_links": {
    "self": { "href": "/books/123" },
    "update": { "href": "/books/123", "method": "PUT" },
    "delete": { "href": "/books/123", "method": "DELETE" }
  }
}
  • Benefit: Clients can follow these links to navigate or perform actions without knowing endpoint structure in advance.

8. HTTP Caching Concepts

  • HTTP provides headers to control how responses are cached by browsers, proxies, and CDNs.
  • Main ones include:
    • Cache-Control
    • ETag / If-None-Match
    • Last-Modified / If-Modified-Since
  • Benefits: Improves performance, reduces bandwidth, and lowers server load.

  • Purpose: Specifies caching rules for browsers and intermediaries (like CDNs).
  • Common directives:
    • max-age=3600 → Cache for 3600 seconds (1 hour).
    • no-cache → Must validate with server before using cached copy.
    • no-store → Don’t store at all (e.g., sensitive data).
    • public → Can be cached by anyone.
    • private → Cache only for a single user (browser only).
  • Example: Cache-Control: public, max-age=86400 → Publicly cacheable for 1 day.

  • ETag: A unique identifier (hash or version) for a specific version of a resource.
  • Sent in the response header by the server: 
    ETag: "abc123"
  • If-None-Match: Sent by the client on the next request to check if the resource changed: 
    If-None-Match: "abc123"
  • If unchanged → server returns 304 Not Modified (no body sent).
  • Benefit: Saves bandwidth by only sending data when it’s changed.

  • Last-Modified: Server sends the last modified date of the resource: 
    Last-Modified: Wed, 07 Aug 2024 12:34:56 GMT
  • If-Modified-Since: Client sends this date in the next request: 
    If-Modified-Since: Wed, 07 Aug 2024 12:34:56 GMT
  • If not changed → 304 Not Modified.
  • Key difference vs ETag:
    • Last-Modified is date-based, ETag is version/hash-based.
    • ETag is more precise (good for small changes).

  • Client-side caching:
    • Happens in the browser.
    • Uses Cache-Control, ETag, Last-Modified.
    • Improves page load speed for the end-user.
  • Server-side caching:
    • Caches data at the server or application layer (e.g., Redis, Memcached).
    • Reduces database calls and processing time.
    • Used for dynamic data that’s expensive to compute.

  • CDN (Content Delivery Network): A network of servers worldwide that caches static and sometimes dynamic content.
  • Delivers content from the nearest location to the user, reducing latency.
  • How it works:
    • CDNs respect HTTP cache headers (Cache-Control, ETag, etc.).
    • Can have their own TTL (Time-To-Live) rules.
  • Example: Images, videos, CSS, JS served via Cloudflare, AWS CloudFront, Akamai, etc.

9. Rate Limiting & Throttling

  • What it is:
    • Rate limiting controls how many requests a client can make in a given time period.
    • Throttling slows down request processing after a limit is reached.
  • Why it’s needed:
    • Prevent abuse (DDoS attacks, brute-force logins, API misuse).
    • Ensure fair usage across all clients.
    • Protect backend resources from overload.
  • Example: Allow 100 requests per minute per user — excess requests get a 429 Too Many Requests response.

  • APIs often return headers so clients know their usage limits.
  • Common headers:
    • X-RateLimit-Limit → Max requests allowed in the time window.
    • X-RateLimit-Remaining → Requests left in the current window.
    • X-RateLimit-Reset → Timestamp when the limit resets.
  • Example response headers: 
    X-RateLimit-Limit: 100
            X-RateLimit-Remaining: 20
            X-RateLimit-Reset: 1691577600
  • This helps clients adapt request frequency to avoid hitting the limit.

  • Spring Boot: Bucket4j / Redis
    • Bucket4j: Java library implementing the token bucket algorithm.
    • Works in-memory or with distributed stores like Redis.
    • Example usage: Define capacity (e.g., 100 tokens) and refill rate (e.g., every minute). Each request consumes a token; if no tokens left → reject request.
    • Redis + Bucket4j: Allows rate limiting across multiple servers in a cluster. Good for microservices or load-balanced setups.
  • NestJS: @Throttle() (NestJS Throttler)
    • Built-in NestJS Throttler package.
    • Decorator-based approach:
      • @Throttle(10, 60) // 10 requests per 60 seconds
@Get()
getData() {
  return 'Some data';
}
    • Global throttling can be configured via ThrottlerModule in app.module.ts.
    • Works in-memory or can be extended with Redis for distributed rate limiting.

10. OpenAPI / Swagger

  • OpenAPI:
    • A standard specification for describing REST APIs in JSON or YAML.
    • Defines endpoints, request/response formats, authentication, etc.
  • Swagger:
    • A set of tools (UI, editor, codegen) for working with OpenAPI specs.
    • Swagger UI: Interactive API documentation where you can try endpoints directly in the browser.
  • Why it matters:
    • Improves developer onboarding.
    • Makes APIs self-explanatory.
    • Reduces miscommunication between backend and frontend teams.

  • springdoc-openapi automatically generates OpenAPI docs from Spring Boot annotations.
  • Setup: Add the dependency in pom.xml or build.gradle.
  • Example usage: 
    @Operation(summary = "Get a book by ID")
@GetMapping("/books/{id}")
public Book getBook(@PathVariable Long id) { ... }
    → Appears in Swagger UI with description and parameters.
  • Default URL: 
    http://localhost:8080/swagger-ui.html

  • Official NestJS package to generate OpenAPI specs from decorators.
  • Setup: Install @nestjs/swagger and swagger-ui-express.
  • Example usage: 
    @ApiTags('books')
@ApiOperation({ summary: 'Get a book by ID' })
@Get(':id')
getBook(@Param('id') id: string) { ... }
  • Generates docs at /api or configured path.
  • Works with DTO classes to document request bodies automatically.

  • Good documentation includes real request/response examples for clarity.
  • Why: Helps developers understand input/output formats without guessing.
  • Example (Swagger): 
    {
  "id": 1,
  "title": "REST API Design",
  "author": "Jane Doe"
}

  • Benefit: Docs stay up-to-date automatically when code changes.
  • How it works:
    • Annotate controllers, DTOs, and methods with metadata.
    • The tool scans the codebase and produces OpenAPI spec.
  • Advantage: Avoids manually maintaining separate documentation files.

12. API Security Best Practices

  • What it is: Use HTTPS (TLS/SSL encryption) for all API communication.
  • Why:
    • Encrypts data in transit to prevent eavesdropping or tampering.
    • Protects sensitive information like passwords, tokens, and personal data.
    • Best practice: Redirect all HTTP traffic to HTTPS and use modern TLS configurations.

  • What it is: Validate all incoming data from clients before processing.
  • Why:
    • Prevents malformed or malicious inputs from breaking the system.
  • Examples:
    • Check data types (string, number).
    • Validate lengths and formats (email regex, max length).
    • Reject unexpected parameters.

  • What it is: SQL injection happens when attackers inject SQL commands into input fields.
  • Solution: Always use prepared statements or ORM query builders (never string concatenation).
  • Example: 
    String sql = "SELECT * FROM users WHERE id = ?";
            preparedStatement.setInt(1, userId);
  • Benefit: Automatically escapes dangerous characters.

  • XSS (Cross-Site Scripting):
    • Happens when malicious JavaScript is injected into a web page.
    • Prevention:
      • Escape HTML output.
      • Use frameworks with built-in sanitization (e.g., React’s auto-escaping).
  • CSRF (Cross-Site Request Forgery):
    • Tricks a logged-in user into performing actions without their consent.
    • Prevention:
      • Use CSRF tokens in forms and API requests.
      • Validate the Origin/Referer headers.

  • Rule: Never send confidential details (passwords, secret keys, internal IDs) in API responses.
  • Examples:
  • Bad: 
    {
  "username": "john",
  "password": "123456"
}
  • Good: 
    {
  "username": "john"
}

  • What it is: Restrict the maximum size of request bodies and responses.
  • Why:
    • Prevents denial-of-service (DoS) attacks using huge payloads.
    • Reduces memory and bandwidth usage.
  • Example:
    • Spring Boot: spring.servlet.multipart.max-request-size=2MB
    • NestJS: Configure body-parser limit in main.ts.

    fetch('/api/books')
.then(res => res.json())
.then(data => console.log(data))
.catch(err => console.error(err));
    Pros: No extra dependency, widely supported.
    Cons: Verbose for complex requests (interceptors, timeouts).
  • Axios: Popular third-party HTTP client. 
    import axios from 'axios';
            
axios.get('/api/books')
  .then(res => console.log(res.data))
  .catch(err => console.error(err));
    Pros:
    • Simpler syntax.
    • Interceptors for auth tokens.
    • Automatic JSON parsing.

Best practice in React: Use state variables to track loading and error states.

const [loading, setLoading] = useState(false);
const [error, setError] = useState(null);
const [data, setData] = useState([]);
            
useEffect(() => {
  setLoading(true);
  fetch('/api/books')
    .then(res => res.json())
    .then(setData)
    .catch(setError)
    .finally(() => setLoading(false));
}, []);

Improves UX by showing spinners, retry options, or error messages.

  • What is CORS: Security feature that controls cross-origin requests.
  • Development example: React app on localhost:3000 calling API on localhost:8080 requires backend CORS enabled.
  • Spring Boot example: 
    @CrossOrigin(origins = "http://localhost:3000")
@RestController
public class BookController { ... }
  • NestJS example: 
    app.enableCors({ origin: 'http://localhost:3000' });

  • Token-based authentication: Store JWT/OAuth tokens securely (preferably in HTTP-only cookies to prevent XSS).
  • Send token with requests:
  • Fetch API example: 
    fetch('/api/books', {

});
  • Axios example: 
    axios.get('/api/books', {
  headers: { Authorization: `Bearer ${token}` }
});
  • Best practice: Avoid storing tokens in localStorage if possible (due to XSS risk). Use secure cookies with SameSite and Secure flags.

  • Example:

  • Download: Return file as binary stream with Content-Disposition: attachment.
  • Example: 
    GET /files/report.pdf

  • What it is: Server-to-server callbacks triggered by events.
  • Flow:
    • Client registers a URL.

  • Best practices:
    • Include event signatures for verification.
    • Retry failed deliveries.

  • When to use: For operations that take a long time (video processing, large reports).
  • Pattern: Client sends request → server returns 202 Accepted with status URL → client polls status endpoint until done.
  • Example: 
    POST /reports → 202 Accepted
Location: /jobs/456
            
GET /jobs/456 → { "status": "in-progress" }

  • What it is: Single entry point for multiple backend services.
  • Responsibilities:
    • Routing requests to microservices.

    • Request/response transformation.
  • Examples: AWS API Gateway, Kong, Nginx, Zuul.
  • Benefits:
    • Simplifies frontend communication.
    • Adds cross-cutting features without modifying backend services.

15. REST API in Spring Boot & NestJS

  • @RestController: Marks class as REST controller (returns JSON by default).
  • @GetMapping, @PostMapping, @PutMapping, @DeleteMapping: Map HTTP methods to Java methods.
  • Example: 
    @RestController
@RequestMapping("/books")
public class BookController {
    @GetMapping("/{id}")
    public Book getBook(@PathVariable Long id) { ... }
}
  • DTOs & validation (@Valid): Define request/response structure with validation annotations. 
    public class BookDTO {
    @NotBlank
    private String title;
    @Min(1)
    private double price;
}
            
@PostMapping
public ResponseEntity createBook(@Valid @RequestBody BookDTO book) { ... }
  • ResponseEntity: Wraps HTTP responses with status, headers, body. 
    return ResponseEntity
.status(HttpStatus.CREATED)
.body(book);
  • Exception handling with @ControllerAdvice: Centralized error handling. 
    @ControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity handleNotFound(ResourceNotFoundException ex) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(ex.getMessage());
    }
}

  • @Controller, @Get, @Post, @Put, @Delete: Mark class & map HTTP methods. 
    @Controller('books')
export class BookController {
  @Get(':id')
  getBook(@Param('id') id: string) { ... }
}
  • DTO validation with class-validator: Use decorators to validate DTOs. 
    export class CreateBookDto {
   @IsNotEmpty()
  title: string;
  @IsPositive()
  price: number;
}
            
@Post()
createBook(@Body() dto: CreateBookDto) { ... }

    Enable ValidationPipe in main.ts for auto-validation.

  • Exception filters: Catch & customize exceptions globally or per-controller. 
    @Catch(HttpException)
export class HttpExceptionFilter implements ExceptionFilter {
  catch(exception: HttpException, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse();
    const status = exception.getStatus();
    response.status(status).json({
      statusCode: status,
      message: exception.message,
    });
  }
}
  • Swagger docs with decorators: Auto-generate OpenAPI docs.
