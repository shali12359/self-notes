1. SQL Basics

Progress:

  • SQL (Structured Query Language) is a standard language used to store, manipulate, and retrieve data from databases.
  • Divided into categories
Category Full Form Purpose Examples
DDL Data Definition Language Defines and manages structure of database objects CREATE, ALTER, DROP, TRUNCATE
DML Data Manipulation Language Handles the data inside tables SELECT, INSERT, UPDATE, DELETE
DCL Data Control Language Controls access and permissions GRANT, REVOKE
TCL Transaction Control Language Manages transactions in a DB COMMIT, ROLLBACK, SAVEPOINT

  • Case-insensitive keywords: SELECT, FROM, WHERE can be written as select, from, where.
  • End statements with ; (some systems require it).
  • String literals → use single quotes 'John' not double quotes (double quotes are for identifiers in some DBs).
  • Identifiers (table/column names) → avoid reserved words, can use underscores (e.g., user_name).
  • Whitespace → ignored, but formatting improves readability.

  • Database → The container for all data (like a big warehouse).
  • Schema → A logical grouping of tables and objects inside a database (like sections in a warehouse).
  • In some systems (MySQL), schema = database.
  • Table → The actual structure holding rows & columns (like shelves storing items).

  • Common SQL data type categories:
    • Numeric — INT (whole numbers), DECIMAL(10,2) (fixed-point), FLOAT (approximate values)
    • Text — CHAR(n) (fixed length), VARCHAR(n) (variable length), TEXT (large text)
    • Date/Time — DATE, TIME, DATETIME, TIMESTAMP
    • Boolean — BOOLEAN or BIT (true/false — stored as 1 or 0 internally in some DBs)

  • NULL = Unknown / Missing value
  • It’s not 0 and not '' (empty string).
  • Comparisons with NULL always return UNKNOWN, not true or false.
  • Use IS NULL or IS NOT NULL instead of =.

  • A) Arithmetic Operators
    • + — Example: 2 + 3 → Result: 5
    • - — Example: 5 - 2 → Result: 3
    • * — Example: 4 * 2 → Result: 8
    • / — Example: 10 / 2 → Result: 5
  • B) Comparison Operators
    • = — equal
    • != or <> — not equal
    • < — less than
    • > — greater than
    • <= — less than or equal
    • >= — greater than or equal
  • C) Logical Operators
    • AND — both conditions must be true
    • OR — at least one condition must be true
    • NOT — reverses the result
    • Example: 
      SELECT * FROM products WHERE price > 100 AND stock > 10;
  • D) Special Operators
    • BETWEEN — Checks if a value is within a range (inclusive). 
      WHERE price BETWEEN 100 AND 500;
    • IN — Matches against a list of values. 
      WHERE country IN ('USA', 'Canada', 'UK');
    • LIKE — Pattern matching with wildcards: % (any number of chars), _ (single char). 
      WHERE name LIKE 'J%n'; -- John, Jason
    • IS NULL — Checks if value is NULL. 
      WHERE manager_id IS NULL;

2. Data Definition Language (DDL)

Progress:

  • Purpose: DDL commands define or modify the structure of database objects (tables, schemas, indexes, etc.).
  • Key points:
    • They work on schemas/structure, not the data inside.
    • They are auto-committed in most databases (changes can’t be rolled back easily).
  • Examples: CREATE, ALTER, DROP, TRUNCATE

  • Purpose: Creates a new table with specified columns, data types, and constraints.
  • Syntax: 
    CREATE TABLE employees (
    id INT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    salary DECIMAL(10,2),
    hire_date DATE);
  • Key notes:
    • Define data type for each column.
    • Use constraints like PRIMARY KEY, NOT NULL, UNIQUE, FOREIGN KEY.

  • Purpose: Changes the structure of an existing table.
  • a) Add a column 
    ALTER TABLE employees ADD department VARCHAR(30);
  • b) Modify (change type or constraints) 
    ALTER TABLE employees MODIFY salary DECIMAL(12,2);
  • c) Drop a column 
    ALTER TABLE employees DROP COLUMN department;

  • Purpose: Deletes the table structure and all its data permanently.
  • Syntax: 
    DROP TABLE employees;
  • Key notes:
    • Cannot be rolled back in most DBs.
    • Removes indexes, constraints, and triggers associated with the table.

  • Purpose: Removes all rows from a table but keeps the structure.
  • Syntax: 
    TRUNCATE TABLE employees;
  • Differences from DELETE:
    • TRUNCATE DELETE
    Removes all rows instantly Can remove specific rows with WHERE
    Cannot use WHERE Can use WHERE
    Resets auto-increment counter (in most DBs) Does not reset
    Faster, minimal logging Slower, logs each row
    DDL command DML command

3. Data Manipulation Language (DML)

  • Purpose: DML commands manipulate the data stored inside database tables.
  • Key points:
    • They operate on data records, not the database structure.
    • DML changes can be rolled back if used within transactions.
    • Commonly used in applications to insert, update, or delete data.
  • Examples: INSERT, UPDATE, DELETE, SELECT

  • Purpose: Adds new rows into a table.
  • Single row insert 
    INSERT INTO table_name (column1, column2) VALUES (value1, value2);
  • Multiple rows insert 
    INSERT INTO table_name (column1, column2) VALUES (value1, value2), (value3, value4), (value5, value6);

  • Purpose: Modifies existing rows in a table.
  • With WHERE clause (safe, updates specific rows) 
    UPDATE table_name SET column1 = value1 WHERE condition;
  • Without WHERE clause (updates all rows) 
    UPDATE table_name SET column1 = value1;

  • Purpose: Removes rows from a table.
  • With WHERE clause (deletes specific rows) 
    DELETE FROM table_name WHERE condition;
  • Without WHERE clause (deletes all rows) 
    DELETE FROM table_name;

4. Basic Queries

  • Purpose: SELECT is used to retrieve data from a database table.
  • You can select one column or multiple columns.
  • Example:
    • Single column: SELECT name FROM employees;
    • Multiple columns: SELECT name, salary FROM employees;

  • Aliases give temporary names to columns or expressions in the result for better readability.
  • The keyword AS is used to assign an alias.
  • Example: SELECT name AS employee_name FROM employees;
  • The alias only affects the output, not the actual column name in the database.

  • The WHERE clause filters rows based on a condition.
  • Only rows meeting the condition will be returned.
  • Example: SELECT * FROM employees WHERE salary > 50000;
  • You can use operators like =, >, <, IN, LIKE, etc., in conditions.

  • ORDER BY sorts the result set by one or more columns.
  • ASC: Ascending order (default).
  • DESC: Descending order.
  • Example: SELECT * FROM employees ORDER BY salary DESC; (highest salary first)

  • Limits how many rows are returned by the query.
  • In MySQL/PostgreSQL use LIMIT.
  • In SQL Server use TOP.
  • Example:
    • MySQL: SELECT * FROM employees LIMIT 10; (first 10 rows)
    • SQL Server: SELECT TOP 10 * FROM employees; (first 10 rows)

5. Aggregate Functions

  • Perform calculations on a set of values and return a single value. They are often used in SQL queries to summarize data.

  • Purpose: Counts the number of rows or non-null values in a column.
  • COUNT(*) counts all rows including NULLs.
  • COUNT(column_name) counts rows where the column is NOT NULL.
  • Example:
    • SELECT COUNT(*) FROM employees; – counts all employees.
    • SELECT COUNT(salary) FROM employees; – counts employees with a salary value.

  • Purpose: Adds up all numeric values in a column.
  • Only works with numeric data types.
  • Example: SELECT SUM(salary) FROM employees; – total of all salaries.

  • Purpose: Calculates the average (mean) of numeric values in a column.
  • Ignores NULL values.
  • Example: SELECT AVG(salary) FROM employees; – average salary.

  • Purpose: Return the smallest (MIN()) or largest (MAX()) value in a column.
  • Example:
    • SELECT MIN(salary) FROM employees; – lowest salary.
    • SELECT MAX(salary) FROM employees; – highest salary.

  • Purpose: Groups rows by one or more columns so aggregates can be calculated per group.
  • Every selected column must be either in GROUP BY or aggregated.
  • Example:
    • SELECT department, AVG(salary) FROM employees GROUP BY department; – average salary per department.

  • Purpose: Filters groups after aggregation, based on aggregate conditions.
  • WHERE filters rows before grouping; HAVING filters groups after.
  • Example:
    • SELECT department, AVG(salary) FROM employees GROUP BY department HAVING AVG(salary) > 50000; – only departments with average salary above 50,000.

6. Joins

  • What it does: Returns only rows where there is a match between two tables based on the join condition.
  • Example: Joining Employees and Departments on DepartmentID returns only employees who belong to a department.
  • Summary: Returns matching rows only.

  • What it does: Returns all rows from the left table, and matched rows from the right table. If no match, right table columns show NULL.
  • Example: LEFT JOIN Employees to Departments returns all employees, even if they don’t belong to a department.
  • Summary: All left table rows + matching right table rows (or NULL).

  • What it does: Returns all rows from the right table, and matched rows from the left table. If no match, left table columns show NULL.
  • Example: RIGHT JOIN Employees to Departments returns all departments, even if no employees belong to them.
  • Summary: All right table rows + matching left table rows (or NULL).

  • What it does: Returns all rows where there is a match in either left or right table. Non-matching rows appear with NULL in the other table’s columns.
  • Example: Combines results of LEFT JOIN and RIGHT JOIN, showing all employees and all departments, matched or not.
  • Summary: All rows from both tables + matches; unmatched parts filled with NULL.

  • What it does: Returns the Cartesian product of the two tables — every row of the first table joined with every row of the second table.
  • Example: If Employees has 3 rows and Departments has 2 rows, result has 3 × 2 = 6 rows.
  • Summary: All possible combinations of rows (no join condition).

  • What it does: A table joins with itself to compare rows within the same table. Uses table aliases to differentiate.
  • Example: Find pairs of employees where one is a manager of another within the same Employees table.
  • Summary: Join a table to itself to find relationships within the same table.

  • ON: Specify any condition to match rows, usually column equality. Requires qualifying column names if ambiguous.
  • USING: Shortcut for joining on columns with the same name in both tables. No need to qualify columns. Only works with equality.
  • Example ON: ON A.id = B.id
  • Example USING: USING (id)
  • Summary: ON is more flexible; USING is simpler but limited.

7. Subqueries

  • Definition: A subquery that returns exactly one row of results, used with operators like =, <, >, etc.
  • Example: 
    SELECT first_name, salary
FROM employees
WHERE salary > (
    SELECT salary
    FROM employees
    WHERE first_name = 'John');
  • Summary: Compares a value from the outer query to a single value returned by the inner query.

  • Definition: A subquery that returns multiple rows, used with operators like IN, ANY, and ALL.
  • IN Example: 
    SELECT first_name
FROM employees
WHERE department_id IN (
    SELECT department_id
    FROM departments
    WHERE location_id = 1700);
  • ANY Example: 
    SELECT first_name, salary
FROM employees
WHERE salary > ANY (
    SELECT salary
    FROM employees
    WHERE department_id = 50);
  • ALL Example: 
    SELECT first_name, salary
FROM employees
WHERE salary > ALL (
    SELECT salary
    FROM employees
    WHERE department_id = 50);
  • Summary:
    • IN: Matches any value in the list.
    • ANY: Compares to at least one value from the list.
    • ALL: Compares to every value from the list.

  • Definition: A subquery that depends on values from the outer query, executing once for each outer row.
  • Example: 
    SELECT e.first_name, e.salary
FROM employees e
WHERE e.salary > (
    SELECT AVG(salary)
    FROM employees
    WHERE department_id = e.department_id);
  • Summary: Useful for row-by-row comparisons but can be slower due to multiple executions.

  • In SELECT: Used to add computed values. 
    SELECT first_name,
    (SELECT department_name
    FROM departments
    WHERE department_id = e.department_id) AS department
FROM employees e;
  • In FROM: Used as a derived table. 
    SELECT dept_id, avg_salary
FROM (
    SELECT department_id AS dept_id, AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department_id
) dept_avg
WHERE avg_salary > 5000;
  • In WHERE: Used to filter rows. 
    SELECT first_name
FROM employees
WHERE department_id IN (
    SELECT department_id
    FROM departments
    WHERE location_id = 1700);

8. Set Operations

  • Definition: Combines results of two queries. UNION removes duplicates, while UNION ALL keeps them.
  • Performance: UNION ALL is faster since it doesn’t remove duplicates.
  • Example:
    • SELECT city FROM customers
UNION
SELECT city FROM suppliers;
                                        
SELECT city FROM customers
UNION ALL
SELECT city FROM suppliers;

  • Definition: Returns only rows present in both queries (common rows).
  • Duplicates: Automatically removed.
  • Example:
    • SELECT city FROM customers

  • Definition: Returns rows from the first query that are not in the second query.

  • Duplicates: Automatically removed.
  • Example:

9. Constraints

  • Definition: A column (or set of columns) that uniquely identifies each row in a table.
  • Key Points: Cannot have duplicate values or NULL, only one primary key per table, can be composite.
  • Example: 
    CREATE TABLE Users (
    user_id INT PRIMARY KEY,
    username VARCHAR(50));
  • Here, user_id uniquely identifies each user.

  • Definition: A column in one table that refers to the PRIMARY KEY in another table.
  • Key Points: Ensures referential integrity, prevents invalid references, supports ON DELETE/ON UPDATE rules.
  • Example: 
    CREATE TABLE Orders (
    order_id INT PRIMARY KEY,
    user_id INT,
    FOREIGN KEY (user_id) REFERENCES Users(user_id));
  • Here, every user_id in Orders must exist in Users.

  • Definition: Ensures all values in a column are different.
  • Key Points: Allows one NULL (in most DBs), multiple unique constraints possible.
  • Example: 
    CREATE TABLE Employees (
    emp_id INT PRIMARY KEY,
    email VARCHAR(100) UNIQUE);
  • Here, no two employees can have the same email.

  • Definition: Ensures a column must have a value, cannot be NULL.
  • Example: 
    CREATE TABLE Products (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL);
  • Every product must have a name.

  • Definition: Ensures values meet a condition.
  • Example: 
    CREATE TABLE Accounts (
    account_id INT PRIMARY KEY,
    balance DECIMAL(10,2) CHECK (balance >= 0));
  • This prevents negative balances.

  • Definition: Assigns a value when none is provided.
  • Example: 
    CREATE TABLE Customers (
    customer_id INT PRIMARY KEY,
    country VARCHAR(50) DEFAULT 'USA');
  • If no country is given, 'USA' is automatically used.

10. Indexes

  • Definition: A database object that stores a sorted copy of selected column(s) along with a pointer to the actual data in the table.
  • Purpose: Speeds up SELECT queries by reducing the number of rows scanned.
  • Trade-off: Uses extra storage and can slow down INSERT, UPDATE, and DELETE.

Feature Clustered Index Non-clustered Index
Definition Sorts and stores the actual data rows in the table according to the index key. Stores a separate structure with the index key and a pointer to the data row.
Data storage Table data is physically ordered by the clustered index. Data remains in the table; index just points to it.
Number allowed Only 1 per table. Many per table (DBMS dependent).
Access speed Faster for range queries. Slightly slower (extra lookup step).
Example Primary key often defaults to clustered index. Index on a name column.

  • Creating an index:

    • Non-clustered index
      • CREATE INDEX idx_customer_name
ON Customers (LastName);
    • Clustered index
      • CREATE CLUSTERED INDEX idx_customer_id
ON Customers (CustomerID);

  • Dropping an index:

    • DROP INDEX idx_customer_name ON Customers;

  • Note: Some DBMS automatically create indexes for primary keys and unique constraints.

  • Use indexes when:

    • Columns are frequently used in WHERE, JOIN, ORDER BY, or GROUP BY.
    • Querying large tables for fast lookups.
    • Column has high cardinality (many unique values).

  • Avoid or limit indexes when:

    • Table is small.
    • Column has low cardinality (few distinct values).
    • Data changes often.
    • You already have many indexes.

  • Positive: Speeds up retrieval, reduces disk I/O.
  • Negative: Slows down writes (INSERT, UPDATE, DELETE), consumes extra storage, unused indexes still cost resources.

  • Key Insight: Indexes are a trade-off — optimize reads at the cost of slower writes and extra storage.

11. Views

  • Definition: A view is a saved SQL query that acts as a virtual table. It doesn’t store data itself; it pulls from base tables when queried.
  • Syntax: 
    CREATE VIEW view_name AS
SELECT column1, column2, ...
FROM table_name
WHERE condition;
  • Example: 
    CREATE VIEW active_employees AS
SELECT employee_id, first_name, last_name, department
FROM employees
WHERE status = 'Active';
  • Benefits:
    • Simplifies complex queries
    • Improves security by limiting data access
    • Ensures consistency in logic

  • Definition: You can update data through a view if it’s updatable — meaning changes can be mapped directly to a single row in a single base table.
  • Updatable View Rules:
    • Must be based on one table
    • No aggregations (SUM, COUNT, etc.)
    • No GROUP BY, DISTINCT, UNION, or calculated columns
  • Example (Valid Update): 
    CREATE VIEW employee_contact AS
SELECT employee_id, email
FROM employees
WHERE status = 'Active';
          
UPDATE employee_contact
SET email = 'new_email@example.com'
WHERE employee_id = 101;
  • Example (Not Updatable): 
    CREATE VIEW department_summary AS
SELECT department_id, COUNT(*) AS total_employees
FROM employees
GROUP BY department_id;
          
-- This fails:
UPDATE department_summary
SET total_employees = 20;
  • Reason for Limitation: The database can’t map changes from aggregated or derived data back to individual rows in the base table.

12. Transactions

  • Definition: A transaction is a sequence of one or more SQL operations executed as a single unit of work.
  • Key Idea: Either all operations succeed (COMMIT) or none take effect (ROLLBACK).
  • Purpose: Helps maintain data integrity and consistency.

  • Purpose: Marks the start of a transaction. All changes are temporary until committed.
  • Syntax: BEGIN; or START TRANSACTION;
  • Example: 
    START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 2;

  • Purpose: Permanently saves all changes made in the transaction.
  • Syntax: COMMIT;
  • Example: 
    COMMIT; -- Finalizes the money transfer

  • Purpose: Undoes all changes in the current transaction.
  • Syntax: ROLLBACK;
  • Example: 
    ROLLBACK; -- Cancels the transfer if one update fails

  • Purpose: Creates a checkpoint in a transaction to roll back partially.
  • Syntax:
    • SAVEPOINT savepoint_name;
    • ROLLBACK TO savepoint_name;
    • RELEASE SAVEPOINT savepoint_name;
  • Example: 
    START TRANSACTION;
UPDATE products SET stock = stock - 5 WHERE id = 1;
SAVEPOINT sp1;
UPDATE products SET stock = stock - 3 WHERE id = 2;
ROLLBACK TO sp1;  -- Undo only the second update
COMMIT;

Property Meaning Example
Atomicity All steps succeed or none do. Money transfer: deduct and add must both succeed.
Consistency Moves DB from one valid state to another, following rules. No negative balances allowed.
Isolation Transactions don't interfere with each other's intermediate states. Two people booking the same seat won't overwrite each other.
Durability Once committed, changes survive system failures. Data after COMMIT remains even after a crash.

13. Performance Optimization

  • Definition: A plan (output of EXPLAIN / EXPLAIN ANALYZE) showing how the DB will execute a query — which indexes it uses, join order, scans, and row estimates.
  • Why it matters: Reveals costly operations (full table scans, expensive joins) so you can target optimizations.
  • Key points: Look for sequential scans, nested loop joins on large sets, and high estimated vs actual row counts.
  • Example action: If plan shows full table scan on a large table, consider adding an index or rewriting the query.
  • Summary: Use execution plans to find where the database spends time and guide fixes.

  • Definition: The process of identifying and diagnosing queries that take too long to run.
  • How to start: Enable slow query logging (e.g., MySQL slow_query_log) or use DB monitoring tools to collect offenders.
  • Things to check: Execution plan, I/O, CPU, row counts, use of indexes, and whether the query returns more data than needed (e.g., SELECT *).
  • Common causes: missing/unused indexes, expensive joins/subqueries, functions on indexed columns, or network/locking issues.
  • Summary: Combine logs, profiling, and execution plans to find root causes and prioritize fixes.

  • Definition: Indexes are data structures that speed up lookups (like a book index) so queries can find rows without scanning the whole table.
  • Best practices: Index columns used in WHERE, JOIN, ORDER BY, and frequently filtered columns. Use composite indexes when multiple columns are commonly queried together (order matters).
  • Trade-offs: Indexes improve read performance but increase storage and slow writes (INSERT/UPDATE/DELETE).
  • Types to know: B-Tree (general), Hash (equality), Full-text (text search), and specialized index types depending on DBMS.
  • Summary: Add well-chosen indexes, avoid over-indexing, and monitor index usage via execution plans and DB stats.

  • Definition: Intentionally duplicating or restructuring data (reducing normalization) to avoid expensive joins and speed up read queries.
  • Why use it: For systems where reads vastly outnumber writes, denormalization reduces join cost and simplifies queries.
  • Trade-offs: More storage, potential data inconsistency, and more complex write/update logic (you must update multiple places).
  • Example: Store customer name in orders to avoid joining orders to customers for every read.
  • Summary: Use denormalization selectively for read-heavy paths; pair with careful update strategies or background jobs to keep data consistent.

14. Database Design Concepts

  • Purpose: Organize data to reduce redundancy and improve integrity.
  • 1NF: Each column holds atomic values; no repeating groups or arrays.
  • 2NF: In 1NF and all non-key columns depend on the entire primary key (no partial dependency).
  • 3NF: In 2NF and no transitive dependencies; non-key columns depend only on the key.
  • BCNF: Stricter than 3NF; every determinant must be a candidate key.

  • Purpose: Intentionally add redundancy to improve read performance.
  • Use Case: High-read workloads where joins are expensive.
  • Trade-off: Faster reads but slower writes and possible inconsistency.

  • Purpose: Visual representation of database structure.
  • Components:
    • Entities: Tables (e.g., Customer, Order).
    • Attributes: Columns (e.g., name, order_date).
    • Relationships: Links between entities.

  • One-to-One: Each record in one table relates to exactly one in another. Example: User ↔ UserProfile.
  • One-to-Many: One record in a table can relate to many in another. Example: Customer → Orders.
  • Many-to-Many: Records in both tables can relate to many in the other; requires a junction table. Example: Students ↔ Courses via StudentCourse.

  • Surrogate Key: Artificial key (e.g., auto-increment ID) with no business meaning.
  • Natural Key: Real-world business identifier (e.g., email, SSN).
Aspect Surrogate Key Natural Key
Stability Stable May change if business rules change
Size Usually small Can be long
Performance Fast joins Slower joins
Meaning None Has business meaning

15. Security & Access Control

  • Definition:
    • Roles: Groups of permissions assigned to users to simplify management.
    • Privileges: Specific rights to perform certain actions in the database (e.g., SELECT, INSERT, UPDATE).
  • Types of privileges:
    • System privileges: Database-wide actions like CREATE USER, BACKUP DATABASE.
    • Object privileges: Access to specific database objects (tables, views, etc.).
  • Example: 
    CREATE ROLE reporting_user;
          GRANT SELECT ON sales TO reporting_user;
          GRANT reporting_user TO john_doe;
  • Key point: Roles make privilege management easier by assigning to multiple users at once.

  • Purpose:
    • GRANT: Give a user or role specific privileges.
    • REVOKE: Remove previously granted privileges.
  • Example: 
    GRANT SELECT, INSERT ON employees TO user1;
          REVOKE INSERT ON employees FROM user1;
  • With grant option: Allows a user to grant the same privilege to others. 
    GRANT SELECT ON employees TO user1 WITH GRANT OPTION;

  • SQL Injection: A vulnerability where attackers inject malicious SQL to manipulate database queries.
  • Example of a vulnerable query: 
    "SELECT * FROM users WHERE username = '" + userInput + "';"
  • Prepared statements solution:
    • Use placeholders instead of string concatenation.
    • Database treats input as data, not executable SQL.
  • Why it works: Prevents malicious input from altering query structure and improves performance.
